## 1. MoAI-ADK란 무엇인가

MoAI-ADK(Agentic Development Kit)는 Claude Code 위에서 동작하는 오픈소스 개발 프레임워크다. "MoAI"는 "모두의 AI(AI for Everyone)"를 의미하며, 한국의 modu-ai 팀이 개발했다.

핵심 철학은 **Hybrid 방법론(TDD + DDD)**이다. 새로운 코드는 TDD(RED → GREEN → REFACTOR)를 적용하고, 기존 코드는 DDD(ANALYZE → PRESERVE → IMPROVE)로 안전하게 개선한다. 요구사항 명세(SPEC)를 먼저 작성하고, 코드와 문서를 항상 동기화 상태로 유지한다.

MoAI-ADK 자체는 별도의 런타임이나 에이전트 엔진이 **아니다**. Claude Code가 제공하는 확장 포인트(CLAUDE.md, 커스텀 커맨드, 에이전트, 스킬, 훅)를 활용하여 개발 방법론을 "코드화"한 **마크다운 프롬프트 + Go 단일 바이너리**다.

## 4. 핵심 워크플로우: 4단계 개발 사이클

MoAI-ADK의 개발 사이클은 4개의 `/moai` 서브커맨드로 구성된다.

### 4.1 전체 흐름

<Mermaid>{`
flowchart LR
    subgraph Project["/moai project\n프로젝트 초기화"]
        P1["설정 생성"]
        P2["언어 설정"]
        P3["Git 전략"]
        P4["DDD/TDD"]
    end
    subgraph Plan["/moai plan\nSPEC 작성"]
        PL1["EARS 형식"]
        PL2["요구사항 명세"]
        PL3["테스트 케이스"]
        PL4["수락 기준"]
    end
    subgraph Run["/moai run\nHybrid 구현"]
        R1["새 코드: TDD"]
        R2["기존 코드: DDD"]
        R3["TRUST 5 검증"]
    end
    subgraph Sync["/moai sync\n문서 동기화"]
        S1["코드↔문서 동기"]
        S2["다이어그램 생성"]
        S3["변경 이력 추적"]
    end

    Project --> Plan
    Plan --> Run
    Run --> Sync
    Sync --> Project
`}</Mermaid>

### 4.2 각 단계 상세

**`/moai project` — 프로젝트 초기화**

프로젝트 문서(product.md, structure.md, tech.md)를 생성한다. 코드베이스를 분석하여 기술 스택, 아키텍처, 디렉토리 구조를 자동으로 문서화한다.

**`/moai plan` — SPEC 작성**

EARS(Easy Approach to Requirements Syntax) 형식으로 구조화된 요구사항 명세를 작성한다. manager-spec 에이전트가 활성화되며, 요구사항을 유비쿼터스/상태 기반/이벤트 기반/선택적/비허용 5가지 패턴으로 분류한다.

**`/moai run` — Hybrid 구현**

Hybrid 방법론을 적용한다:

- **새로운 코드:** RED → GREEN → REFACTOR (TDD)
  - **RED:** 테스트 코드만 작성. 의도적으로 실패해야 함.
  - **GREEN:** 테스트를 통과시키는 최소한의 코드만 작성.
  - **REFACTOR:** 테스트 통과를 유지하면서 코드 품질 개선.

- **기존 코드:** ANALYZE → PRESERVE → IMPROVE (DDD)
  - **ANALYZE:** 기존 코드와 의존성 분석.
  - **PRESERVE:** Characterization 테스트로 현재 동작 보존.
  - **IMPROVE:** 테스트 보호 아래에서 점진적 개선.

각 단계는 TodoWrite로 실시간 추적되며, 한 번에 하나의 태스크만 in_progress 상태를 가진다.

**`/moai sync` — 문서 동기화**

코드와 문서를 자동으로 동기화한다. 다이어그램 생성, 변경 이력 추적, Git 커밋을 포함한다. 커밋 메시지는 방법론 사이클을 반영한다.

```
# TDD 사이클 (새로운 코드)
commit 1: "test: Add authentication integration tests"     (RED)
commit 2: "feat: Implement JWT token service (minimal)"     (GREEN)
commit 3: "refactor: Enhance JWT error handling"            (REFACTOR)

# DDD 사이클 (기존 코드)
commit 1: "test: Add characterization tests for user service" (PRESERVE)
commit 2: "refactor: Extract validation logic"                 (IMPROVE)

commit 4: "merge: Merge SPEC-AUTH-001 to develop"
```

## 5. 에이전트와 스킬 시스템

### 5.1 에이전트 (28개)

MoAI-ADK의 에이전트는 Claude Code의 서브에이전트 시스템을 활용한다. 각 에이전트는 `.claude/agents/moai/` 디렉토리의 마크다운 파일로 정의되며, 역할(role), 집중 영역(focus), 책임(responsibilities)이 명시된다.

| 분류 | 에이전트 | 역할 |
|---|---|---|
| 핵심 워크플로우 (Manager) | manager-spec | EARS 형식 요구사항 설계 |
| | manager-ddd | Domain-Driven Development, ANALYZE-PRESERVE-IMPROVE |
| | manager-tdd | Test-Driven Development, RED-GREEN-REFACTOR |
| | manager-docs | 문서 생성 및 동기화 |
| | manager-git | 버전 관리 자동화 |
| | manager-quality | TRUST 5 품질 게이트 |
| | manager-project | 프로젝트 설정 및 구조 관리 |
| | manager-strategy | 시스템 설계 및 아키텍처 결정 |
| 도메인 전문가 (Expert) | expert-backend | 서버 아키텍처, API 개발 |
| | expert-frontend | UI/컴포넌트 설계 |
| | expert-security | 보안 분석 및 컴플라이언스 |
| | expert-devops | 배포 및 인프라 |
| | expert-performance | 성능 최적화 |
| | expert-debug | 디버깅 및 오류 분석 |
| | expert-testing | 테스트 전략 및 커버리지 |
| | expert-refactoring | 코드 리팩토링 |
| 빌더 (Builder) | builder-agent | 새로운 에이전트 정의 생성 |
| | builder-command | 새로운 슬래시 커맨드 생성 |
| | builder-skill | 새로운 스킬 생성 |
| | builder-plugin | 새로운 플러그인 생성 |
| 팀 (Team) | team-researcher | 읽기 전용 탐색 및 연구 |
| | team-analyst | 요구사항 분석 |
| | team-architect | 아키텍처 설계 |
| | team-designer | UI/UX 설계 |
| | team-backend-dev | 서버 측 구현 |
| | team-frontend-dev | 클라이언트 측 구현 |
| | team-tester | 테스트 작성 |
| | team-quality | TRUST 5 품질 검증 |

에이전트 선택은 자동으로 이루어진다. 사용자 입력을 분석하여 도메인을 감지하고, 복잡도를 평가하여, 적절한 에이전트를 할당한다. Agent Teams 모드에서는 여러 에이전트가 병렬로 협업한다.

### 5.2 스킬 (64+)

스킬은 도메인별 지식을 캡슐화한 마크다운 파일이다. Foundation, Workflow, Domain, Language(16개), Platform, Library, Tool 7개 티어로 분류된다. 스페셜리스트 스킬(Figma, Flutter, Chrome Extension, Electron 등)도 포함된다.

> **전체 스킬 목록 및 사용 시점 가이드**: [moai-skills-reference.md](./moai-skills-reference.md)

### 5.3 훅

6가지 훅 이벤트가 품질 게이트 역할을 한다. Go로 컴파일된 `moai hook` 바이너리가 JSON 프로토콜로 처리한다.

- **SessionStart:** 세션 시작 시 프로젝트 설정 검증, 버전 매칭 확인
- **SessionEnd:** 세션 종료 시 상태 정리, 캐시 동기화
- **PreToolUse:** 도구 실행 전 검증 (SPEC 없이 코드 작성 시도 차단 등)
- **PostToolUse:** 도구 실행 후 검증 (테스트 커버리지 확인 등)
- **RequestStart:** 사용자 요청 시작 시 컨텍스트 준비
- **RequestEnd:** 사용자 요청 완료 시 상태 저장

Hook 래퍼는 Shell 스크립트로 구현되며, JSON을 stdin으로 받아 `moai hook <event>` 바이너리를 호출한다.

## 6. LLM 교체 메커니즘: GLM 사용

MoAI-ADK는 **LLM 교체를 공식 지원**한다. z.ai의 GLM 모델로 전환하면 Claude 모델은 **완전히 대체**되어 한 톨도 사용하지 않는다.

### 6.1 동작 원리

Claude Code는 내부적으로 `ANTHROPIC_BASE_URL`과 `ANTHROPIC_AUTH_TOKEN` 환경 변수로 LLM API 엔드포인트를 결정한다. GLM 전환 시 이 환경 변수를 z.ai의 엔드포인트로 교체한다.

```bash
# GLM 전환이 내부적으로 수행하는 작업 (개념적으로):
export ANTHROPIC_BASE_URL=https://api.z.ai/api/anthropic
export ANTHROPIC_AUTH_TOKEN=YOUR_GLM_API_KEY
export ANTHROPIC_MODEL="glm-4.7"
claude    # Claude Code를 시작하지만, 모든 LLM 호출은 z.ai로 간다
```

z.ai는 Anthropic Messages API와 호환되는 엔드포인트를 제공하기 때문에, Claude Code 입장에서는 "Anthropic API를 호출하는 것 같지만" 실제로는 GLM이 응답한다. Layer 2(Claude Code의 도구 실행 엔진)와 Layer 3(MoAI-ADK의 스킬/커맨드)는 변하지 않는다.

### 6.2 설정 과정

GLM 사용은 환경 변수 설정으로 이루어지며, MoAI-ADK는 GLM 전용 CLI를 제공하지 않는다. 사용자가 직접 환경 변수를 설정해야 한다.

```bash
# 1. z.ai에서 GLM API 키 발급
# https://z.ai 에서 가입 후 API 키 획득

# 2. Claude Code 설정에 환경 변수 추가
# ~/.claude/settings.json 또는 프로젝트의 .claude/settings.json에 추가:
{
  "env": {
    "ANTHROPIC_BASE_URL": "https://api.z.ai/api/anthropic",
    "ANTHROPIC_AUTH_TOKEN": "YOUR_GLM_API_KEY",
    "ANTHROPIC_MODEL": "glm-4.7"
  }
}

# 3. Claude Code 실행 (내부적으로 GLM이 응답)
claude

# 4. MoAI 워크플로우 실행 (동일하게 동작)
> /moai SPEC-AUTH-001
```

### 6.3 하이브리드 전략: Opus + GLM

MoAI-ADK는 "Opus로 설계하고 GLM으로 대량 구현"하는 하이브리드 전략을 권장하며, 이를 통해 비용을 최대 70%까지 절감할 수 있다고 안내한다. 이는 Git Worktree를 통해 세션별로 LLM을 분리하는 방식으로 구현된다.

```
터미널 1 (메인) ─── Claude Opus ─── 아키텍처 설계, 코드 리뷰
  │
  ├── 터미널 2 (feat/auth worktree) ─── GLM 4.7 ─── 구현 작업
  │     $ moai worktree new feat-auth
  │     $ cd .worktrees/feat-auth
  │     $ claude  # GLM 설정 적용
  │     > /moai run SPEC-AUTH-001
  │
  ├── 터미널 3 (feat/pay worktree) ─── GLM 4.7 ─── 구현 작업
  │     $ moai worktree new feat-pay
  │     $ cd .worktrees/feat-pay
  │     $ claude  # GLM 설정 적용
  │     > /moai run SPEC-PAY-002
  │
  └── 터미널 4 (feat/ui worktree) ─── GLM 4.7 ─── 구현 작업
```

**주의사항:** 같은 세션에서 LLM을 변경하면 인증 에러가 발생하여 세션 계속이 어렵다. Git Worktree로 각 SPEC을 완전히 격리하여 독립적인 LLM 설정을 유지하는 것이 권장된다.

### 6.4 비용 비교

|---|---|---|
| 접근 방식 | Claude Code Max 구독 또는 API | GLM Coding Plan |

## 8. 유사 프레임워크 비교

MoAI-ADK와 같이 Claude Code 위에서 동작하는 개발 프레임워크가 2025년 하반기부터 활발하게 등장했다. 모두 Claude Code의 확장 포인트(CLAUDE.md, 커스텀 커맨드, 에이전트, 스킬, 훅)를 활용하여 개발 방법론을 코드화한다는 공통점이 있다.

### 8.1 종합 프레임워크

#### Claude Flow (ruvnet/claude-flow)

멀티 에이전트 스웜 오케스트레이션 플랫폼이다. MCP 프로토콜을 통해 Claude Code와 네이티브 연동되며, 5가지 실행 모드(Autopilot, Ultrapilot, Swarm, Pipeline, Ecomode)를 제공한다. 31개 이상의 스킬, 32개 전문 에이전트를 포함한다. AgentDB로 벡터 검색 기반 학습과 메모리를 지원하며, 플러그인 생태계(Neural Coordination, Quantum Reasoning 등)가 활발하다.

- **MoAI-ADK와의 차이:** MoAI-ADK가 SPEC-First TDD라는 특정 방법론에 집중하는 반면, Claude Flow는 다양한 오케스트레이션 패턴을 범용적으로 제공한다.

#### Metaswarm (dsifry/metaswarm)

18개 에이전트, TDD, 100% 커버리지를 목표로 하는 자기 개선형 오케스트레이션 프레임워크다. Research → Plan → Design Review Gate → Implement → Code Review → PR Creation → PR Shepherd → Closure & Learning의 8단계 워크플로우를 따른다. BEADS(git-native 이슈 트래킹)를 조정 백본으로 사용하며, 프로덕션 멀티테넌트 SaaS에서 수백 개의 자율 PR을 처리한 실전 사례가 있다. 사용자 피드백과 마찰 포인트를 자동 수집하여 지식 베이스를 점진적으로 확장하는 자기 학습 메커니즘이 특징이다.

- **MoAI-ADK와의 차이:** 8단계 파이프라인과 자기 학습이 더 정교하지만, BaaS 스킬이나 LLM 교체 같은 기능은 없다.

#### SPARC Methodology

Specification → Pseudocode → Architecture → Refinement → Completion의 5단계 방법론을 셸 스크립트로 구현한 것이다. London School TDD(목과 행위 테스트), 병렬 BatchTool 오케스트레이션, 자동 린팅/보안 검증을 포함한다. 별도 패키지 설치 없이 셸 스크립트 + CLAUDE.md만으로 동작하므로 도입이 가볍다.

- **MoAI-ADK와의 차이:** 스킬/에이전트 수가 적고 CLI 도구가 없지만, 설치와 이해가 매우 간단하다.

### 8.2 특정 영역 집중형

#### Conductor (wshobson/agents 내 포함)

Claude Code를 프로젝트 관리 도구로 변환하는 워크플로우다. `/conductor:setup`으로 프로젝트 비전을 생성하고, `/conductor:new-track`으로 명세와 구현 계획을 만들고, `/conductor:implement`로 TDD를 실행한다. 전체 저장소에는 112개 에이전트, 146개 스킬, 73개 플러그인이 포함되어 있다.

#### claude-code-workflows (shinpr/claude-code-workflows)

`/implement "기능 설명"` 하나로 요구사항 분석부터 TDD 구현, 리뷰까지 자동 실행되는 프로덕션 워크플로우 플러그인이다. 백엔드와 프론트엔드 플러그인이 분리되어 있으며, "메트로놈(metronome)"이라는 안전장치가 Claude의 벌크 bash 실행을 감지하고 차단한다.

#### TDD Guard / TDD Integration Skill

스킬과 서브에이전트를 결합하여 엄격한 RED-GREEN-REFACTOR를 강제하는 접근법이다. RED/GREEN/REFACTOR 각 단계를 별도 서브에이전트로 분리하여 "컨텍스트 오염" 문제를 해결한다. 하나의 컨텍스트에서 테스트 작성과 구현이 동시에 이루어지면 LLM이 무의식적으로 구현을 고려한 테스트를 작성하기 때문이다.

## 10. 핵심 요약

**MoAI-ADK의 본질은 무엇인가?**

Claude Code 위에 Hybrid 방법론(TDD + DDD)을 강제하는 마크다운 프롬프트 + Go 단일 바이너리 CLI 도구다. 자체 런타임은 없으며, Claude Code의 표준 확장 포인트를 활용한다.

**Go 버전의 주요 개선점은?**

Python 버전 대비 다음과 같은 개선이 이루어졌다:
- 단일 바이너리 배포 (pip + venv 의존성 제거)
- 5ms 미만의 빠른 시작 시간 (~800ms → ~5ms)
- 네이티브 고루틴 기반 동시성 처리
- 컴파일 타임 타입 안정성 보장
- 크로스 플랫폼 사전 빌드 바이너리 제공
- Go로 컴파일된 Hook 시스템 (JSON 프로토콜)

**GLM을 쓰면 Claude 모델은 사용하지 않는가?**

맞다. 환경 변수를 z.ai로 설정하면 API 엔드포인트가 완전히 교체되어 Claude 모델 비용은 0원이 된다. Claude Code(도구 실행 엔진)는 그대로 사용하지만, 두뇌(LLM)만 GLM으로 바뀐다.

**다른 유사 프레임워크와 어떻게 다른가?**

LLM 교체 공식 지원, Go 단일 바이너리 CLI, 9개 BaaS 플랫폼 통합, Agent Teams 지원, 한국어 기본 지원이 MoAI-ADK만의 차별점이다. 방법론적으로는 Hybrid (TDD + DDD)를 자동으로 적용하며, Metaswarm과 가장 유사하고, 범용성은 Claude Flow가, 도입 간편성은 SPARC가 우수하다.
