## 1. 에이전트란 무엇인가

### 1.1 정의

터미널에서 `claude`를 실행하면 시작되는 하나의 Claude Code 세션이 곧 **에이전트(Agent)**다. 비유하자면 여러분 옆에 앉아 있는 시니어 개발자 한 명이다.

이 에이전트는 대화를 통해 지시를 받고, 스스로 판단하여 행동한다.

```
개발자: "사용자 인증 API를 만들어줘"

에이전트의 자율적 행동 흐름:
  1. 프로젝트 구조 파악 (파일 탐색)
  2. 기존 코드 패턴 분석 (Grep, Read)
  3. 구현 계획 수립
  4. 모델/스키마/라우터/테스트 파일 생성 (Write)
  5. 테스트 실행 (Bash)
  6. 실패 시 원인 분석 후 코드 수정
  7. 테스트 통과 확인
  8. 결과 보고
```

기존 AI 코딩 도구와 결정적으로 다른 점은 **행동(Action)을 직접 수행한다**는 것이다. 코드를 "제안"하는 것이 아니라, 실제로 파일을 만들고, 명령어를 실행하고, 그 결과를 보고 다음 행동을 결정한다.

### 1.2 에이전트가 할 수 있는 것

에이전트는 다음 도구(Tools)를 통해 개발 환경과 직접 상호작용한다.

| 도구 | 기능 | 예시 |
|------|------|------|
| Read | 파일 내용 읽기 | 소스 코드, 설정 파일 확인 |
| Write | 새 파일 생성 | 모델, 라우터, 테스트 파일 작성 |
| Edit | 기존 파일 수정 | 버그 수정, 리팩토링 |
| Bash | 쉘 명령 실행 | `pytest`, `npm run build`, `git commit` |
| Glob | 파일 패턴 검색 | `*.py` 파일 목록 조회 |
| Grep | 텍스트 검색 | 특정 함수 사용처 찾기 |
| WebFetch | 웹 페이지 가져오기 | 라이브러리 문서 참조 |
| WebSearch | 웹 검색 | 에러 메시지 해결책 검색 |

### 1.3 컨텍스트 윈도우 — 에이전트의 기억 한계

에이전트에게는 **컨텍스트 윈도우**라는 기억의 한계가 있다. 하나의 세션에서 주고받은 대화, 읽은 파일, 실행한 명령의 결과가 모두 이 윈도우 안에 쌓인다. 윈도우가 가득 차면 초반 내용부터 잊기 시작한다.

<Mermaid>{`
flowchart LR
    subgraph ctx["컨텍스트 윈도우 (예: 200K 토큰)"]
        A["CLAUDE.md\n(항상 유지)"] --> B["대화 1"]
        B --> C["파일 읽기"]
        C --> D["대화 2"]
        D --> E["..."]
    end
    E -->|"가득 차면\n초반부터 밀려남"| A
`}</Mermaid>

이 한계가 서브에이전트와 에이전트 팀이 필요한 근본적인 이유다.

- **서브에이전트**: 별도의 컨텍스트에서 작업하고 결과 요약만 반환 → 메인 세션의 컨텍스트 절약
- **에이전트 팀**: 각 팀원이 자기만의 컨텍스트를 보유 → 전체 작업량 분산

### 1.4 CLAUDE.md — 에이전트에게 주는 프로젝트 브리핑

CLAUDE.md는 프로젝트 루트에 위치하는 마크다운 파일로, 에이전트가 세션을 시작할 때마다 자동으로 읽는다. 새로 합류한 팀원에게 전달하는 "프로젝트 온보딩 문서"와 같다.

```markdown
# 프로젝트: TaskFlow

## 기술 스택
- 백엔드: Python 3.12, FastAPI, SQLAlchemy 2.0
- 프론트엔드: Next.js 15, TypeScript, Tailwind CSS
- DB: PostgreSQL 16

## 코딩 컨벤션
- PEP 8 준수, type hint 필수
- Conventional Commits 형식 커밋
- 변경 전 반드시 git branch 생성

## 금지 사항
- 환경 변수 하드코딩 금지
- any 타입 사용 금지
```

CLAUDE.md에 작성한 규칙은 서브에이전트와 에이전트 팀원도 동일하게 따른다. 프로젝트 전체의 일관성을 유지하는 핵심 수단이다.

## 2. 서브에이전트 — 전문가에게 위임하기

### 2.1 개념

서브에이전트는 메인 에이전트가 특정 작업을 위임하기 위해 불러오는 **일회성 전문가**다.

회사에서 일하다가 법률 검토가 필요하면 법무팀에 요청하고, 결과만 받아보는 것과 같다. 법무팀이 내부적으로 어떤 과정을 거쳤는지 일일이 알 필요 없이, 최종 검토 의견만 받으면 된다.

<Mermaid>{`
flowchart TD
    subgraph main["메인 에이전트 세션"]
        A["인증 API를 구현하고, 코드 리뷰도 받자"] --> B["1. 직접 인증 API 코드 작성"]
        B --> C["2. 리뷰가 필요하겠군"]
        C --> D

        subgraph sub["서브에이전트: 코드 리뷰어 (별도 컨텍스트)"]
            D["보안 취약점 점검\n성능 이슈 확인\n컨벤션 준수 여부 검토"]
            D --> E["결과: 2건 Warning, 1건 Info 발견"]
        end

        E -->|"요약된 결과만 반환"| F["3. 리뷰 결과 반영하여 코드 수정"]
    end
`}</Mermaid>

### 2.2 핵심 특성

서브에이전트를 이해하기 위해 반드시 알아야 할 네 가지 특성이 있다.

**별도의 컨텍스트에서 실행된다**: 서브에이전트는 자기만의 컨텍스트 윈도우를 가진다. 방대한 파일을 분석하거나 긴 로그를 처리하더라도 그 과정의 세부 내용이 메인 세션의 컨텍스트를 소모하지 않는다. 결과 요약만 메인으로 돌아온다.

**순차적으로 실행된다**: 이것이 가장 중요한 포인트다. 서브에이전트는 **병렬로 실행되지 않는다.** 메인 에이전트가 서브에이전트를 호출하면, 그 작업이 완료될 때까지 메인은 대기한다.

```
서브에이전트의 실행 흐름 (순차):

시간 →

메인    ████ 작업 ██ 대기 ████████ 결과받음 ██ 대기 ████████ 결과받음 ████
서브A                █████████████|
서브B                                        █████████████|
                     ↑                       ↑
              서브A 호출                서브B 호출
              (메인 대기)               (메인 대기)
```

**일회성이다**: 서브에이전트는 작업이 끝나면 사라진다. 다음에 같은 서브에이전트를 호출해도 이전 작업의 기억은 없다. 다만, `resume` 명령으로 이전 서브에이전트의 작업을 이어갈 수는 있다.

**중첩할 수 없다**: 서브에이전트가 또 다른 서브에이전트를 호출하는 것은 불가능하다. 이 제한은 무한 위임 루프를 방지하기 위한 것이다.

### 2.3 도구 접근 권한 제한

서브에이전트의 강력한 기능 중 하나는 **사용 가능한 도구를 제한**할 수 있다는 것이다.

| 에이전트 유형 | 권장 도구 | 이유 |
|-------------|----------|------|
| 코드 리뷰어 | Read, Grep, Glob | 읽기만 가능 → 실수로 코드 수정 방지 |
| 리서치 분석가 | Read, Grep, Glob, WebFetch, WebSearch | 정보 수집에 특화 |
| 백엔드 개발자 | Read, Write, Edit, Bash, Glob, Grep | 코드 작성 + 실행 가능 |
| QA 엔지니어 | Read, Write, Edit, Bash, Glob, Grep | 테스트 작성 + 실행 가능 |

읽기 전용 에이전트에게는 Write, Edit, Bash를 빼서 안전하게 운영할 수 있다.

### 2.4 서브에이전트 정의 방법

서브에이전트는 `.claude/agents/` 디렉토리에 마크다운 파일로 정의한다.

```
프로젝트 루트/
└── .claude/
    └── agents/
        ├── backend-developer.md
        ├── frontend-developer.md
        ├── code-reviewer.md
        ├── qa-engineer.md
        └── devops-engineer.md
```

파일 형식은 YAML 프론트매터 + 마크다운 본문이다.

```markdown
{/* .claude/agents/code-reviewer.md */}
---
name: code-reviewer
description: 코드 품질, 보안, 성능을 검토하는 리뷰 전문 에이전트
tools:
  - Read
  - Grep
  - Glob
---

# 코드 리뷰어

## 역할
변경된 코드의 품질, 보안, 성능을 검토하고 이슈를 보고합니다.

## 검토 관점
- 보안: SQL 인젝션, XSS, 인증 우회, 민감 데이터 노출
- 성능: N+1 쿼리, 불필요한 연산, 인덱스 누락
- 품질: 코딩 컨벤션 준수, 에러 처리 완전성, 타입 안전성
- 테스트: 커버리지, 엣지 케이스, 에러 시나리오

## 출력 형식
각 이슈를 다음 형식으로 보고합니다:
- [Critical] 즉시 수정 필요
- [Warning] 수정 권장
- [Info] 개선 제안
```

**프론트매터 필드 설명**

| 필드 | 필수 | 설명 |
|------|------|------|
| `name` | 필수 | 에이전트 식별자 (호출 시 사용) |
| `description` | 필수 | 에이전트 역할 설명 (Claude Code가 적절한 에이전트를 자동 선택할 때 참조) |
| `tools` | 선택 | 사용 가능한 도구 목록 (미지정 시 모든 도구 사용 가능) |

### 2.5 서브에이전트 사용 방법

**방법 1: 명시적으로 호출하기**

```
> code-reviewer 서브에이전트를 사용해서 backend/app/api/auth.py를 리뷰해줘
```

```
> qa-engineer 서브에이전트로 인증 API에 대한 테스트를 작성해줘
```

**방법 2: 작업 흐름 안에서 자연스럽게 요청하기**

```
> 인증 API를 구현한 다음, 서브에이전트로 코드 리뷰를 받고,
> 다른 서브에이전트로 테스트를 작성해줘
```

이렇게 지시하면 Claude Code가 순차적으로 작업 → 리뷰 서브에이전트 호출 → 테스트 서브에이전트 호출 흐름을 자동으로 진행한다.

**방법 3: Claude Code의 자동 판단에 맡기기**

서브에이전트를 명시적으로 요청하지 않아도, Claude Code가 작업의 특성을 보고 알아서 서브에이전트를 활용하는 경우가 있다. 예를 들어 복잡한 코드베이스 분석이 필요한 상황에서, 메인 컨텍스트를 아끼기 위해 자체적으로 서브에이전트를 띄울 수 있다.

**방법 4: `/agents` 슬래시 커맨드 사용하기**

```
> /agents
```

이 명령으로 현재 프로젝트에 정의된 서브에이전트 목록을 확인하고 관리할 수 있다.

### 2.6 서브에이전트를 쓰면 좋은 상황

- **코드 리뷰**: 읽기 전용 에이전트에게 위임, 메인 컨텍스트 보존
- **테스트 작성**: 독립적으로 완결되는 작업
- **대규모 파일 분석**: 방대한 로그나 코드를 분석하고 요약만 받기
- **문서 생성**: API 문서, README 등 독립적 산출물 생성
- **특정 관점의 검토**: 보안 전문가, 성능 전문가 등 역할을 나눈 순차 검토

### 2.7 서브에이전트를 쓰지 않는 게 좋은 상황

- **반복적 대화가 필요한 작업**: 아키텍처 논의처럼 여러 차례 피드백을 주고받아야 하는 경우, 메인 세션에서 직접 진행하는 게 낫다
- **아주 단순한 작업**: 서브에이전트 생성 자체에 오버헤드가 있으므로, 간단한 수정은 메인에서 직접 하는 게 빠르다
- **이전 맥락에 의존하는 작업**: 서브에이전트는 메인 세션의 대화 히스토리를 모른다

## 3. 에이전트 팀 — 진정한 병렬 협업

### 3.1 개념

에이전트 팀은 여러 Claude Code 인스턴스가 **동시에** 작업하면서 **서로 소통**하는 기능이다. 같은 사무실에서 함께 일하는 동료 개발자 여러 명이라고 생각하면 된다.

서브에이전트와 결정적으로 다른 두 가지가 있다.

**첫째, 진정한 병렬 실행이다.** 서브에이전트는 역할만 분리했을 뿐 실제로는 한 번에 하나씩 순차 실행된다. 에이전트 팀은 여러 인스턴스가 동시에 돌아간다.

```
서브에이전트 (순차):

시간 →
메인    ████ 지시 ── 대기 ──── 결과 ── 지시 ── 대기 ──── 결과 ████
서브A              ██████████|
서브B                                       ██████████|

총 소요 시간 = 서브A 시간 + 서브B 시간
────────────────────────────────────────────────

에이전트 팀 (병렬):

시간 →
팀 리드 ████ 배정 ─────────────────── 결과 종합 ████
팀원A         ██████████████████████|
팀원B         ██████████████████████|
팀원C         █████████████████████████|

총 소요 시간 ≈ 가장 오래 걸리는 팀원의 시간
```

**둘째, 팀원 간 소통이 가능하다.** 서브에이전트는 메인에게만 결과를 보고하고, 다른 서브에이전트의 존재를 모른다. 에이전트 팀의 팀원들은 서로에게 직접 메시지를 보낼 수 있다.

**셋째, 사전 정의 파일이 필요 없다.** 서브에이전트는 `.claude/agents/` 디렉토리에 마크다운 파일을 미리 작성해야 사용할 수 있다(섹션 2.4 참조). 반면 에이전트 팀은 환경 변수만 활성화하면, 자연어 프롬프트 한 번으로 팀원의 역할·범위·도구를 즉석에서 정의하고 바로 실행할 수 있다. 반복적으로 같은 팀 구성을 사용한다면 서브에이전트 파일로 정의해두는 것이 효율적이지만, 그것은 선택사항이지 필수가 아니다.

```
서브에이전트의 소통 구조:

  메인 ← 결과 ← 서브A
  메인 ← 결과 ← 서브B
  (서브A와 서브B는 서로 모름)

에이전트 팀의 소통 구조:

  팀원A ──→ 팀원B: "백엔드 스키마 변경 완료, 타입 업데이트 해줘"
  팀원B ──→ 팀원C: "프론트 컴포넌트 수정했으니 테스트 확인 부탁"
  팀원C ──→ 팀 리드: "전체 테스트 통과"
```

### 3.2 에이전트 팀의 구조

에이전트 팀은 네 가지 요소로 구성된다.

<Mermaid>{`
flowchart TD
    subgraph team["에이전트 팀"]
        TL["팀 리드\nClaude Code를 시작한 메인 세션\n팀 생성, 작업 배정, 결과 종합"]

        subgraph tasks["공유 태스크 리스트"]
            T1["태스크1\n할당됨"]
            T2["태스크2\n진행중"]
            T3["태스크3\n진행중"]
            T4["태스크4\n대기중"]
        end

        TL --> tasks

        tasks --> MA["팀원 A\n독립 컨텍스트"]
        tasks --> MB["팀원 B\n독립 컨텍스트"]
        tasks --> MC["팀원 C\n독립 컨텍스트"]

        MA <-->|"상호 메시지 교환"| MB
        MB <-->|"상호 메시지 교환"| MC
        MA <-->|"상호 메시지 교환"| MC
    end
`}</Mermaid>

**팀 리드**: `claude`를 실행한 메인 세션. 팀을 생성하고, 작업을 배정하고, 최종 결과를 종합한다. 팀 리드는 고정이며 변경할 수 없다.

**팀원**: 팀 리드가 생성한 독립적인 Claude Code 인스턴스. 각자 고유한 컨텍스트 윈도우를 가진다. 프로젝트의 CLAUDE.md와 MCP 서버 설정을 동일하게 로드한다.

**공유 태스크 리스트**: 모든 팀원이 볼 수 있는 작업 목록. 팀원은 사용 가능한 태스크를 확인하고 가져갈(claim) 수 있다.

**메시지 시스템**: 팀원 간 직접 메시지를 주고받는 채널. 작업 완료 알림, 의존 관계 전달, 공유 결정 사항 등을 교환한다.

### 3.3 활성화 방법

에이전트 팀은 현재 실험적(experimental) 기능이며, 명시적으로 활성화해야 한다.

**방법 1: 환경 변수 (일시적)**

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
claude
```

**방법 2: settings.json (영구적)**

```json
// ~/.claude/settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

활성화 확인:

```bash
echo $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS
# 출력: 1
```

### 3.4 사용 방법

에이전트 팀은 자연어로 지시한다. 서브에이전트처럼 `.claude/agents/` 에 마크다운 파일을 미리 작성할 필요가 없다. 이 차이가 중요하다.

```
서브에이전트 사용 전 필요한 준비:
  1. .claude/agents/code-reviewer.md 작성  ← 파일 정의 필수
  2. name, description, tools 등 프론트매터 설정
  3. 시스템 프롬프트 작성
  4. Claude Code에서 호출

에이전트 팀 사용 전 필요한 준비:
  1. CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 설정  ← 환경 변수만 활성화
  2. Claude Code에서 자연어로 팀 구성 요청        ← 즉시 시작
```

Claude Code에게 팀 구성과 작업을 설명하면, 팀 리더가 자동으로 팀원을 생성하고 역할을 부여한다. Claude가 스스로 작업의 특성을 보고 팀 생성을 제안하는 경우도 있으며, 이때 사용자가 확인한 후 진행된다.

**예시 1: 코드 리뷰 팀**

처음 에이전트 팀을 사용한다면 읽기 전용 작업부터 시작하는 것을 권장한다.

```
> 에이전트 팀을 만들어서 이 PR을 리뷰해줘.
> 세 명의 리뷰어를 투입해:
> - 보안 관점 리뷰어
> - 성능 관점 리뷰어  
> - 테스트 커버리지 리뷰어
> 각 리뷰어의 결과를 종합해서 우선순위별로 정리해줘.
```

**예시 2: 기능 개발 팀**

```
> 에이전트 팀으로 태스크 마감일 기능을 구현해줘.
> - 백엔드 팀원: 모델 + 마이그레이션 + API 수정
> - 프론트엔드 팀원: UI 컴포넌트 업데이트
> - QA 팀원: 테스트 작성 및 실행
> 백엔드가 스키마를 완료하면 프론트엔드에게 알려줘.
```

**예시 3: 디버깅 팀**

```
> 로그인 실패 버그를 디버깅하기 위해 에이전트 팀을 만들어줘.
> - 팀원 1: 프론트엔드 인증 플로우 추적
> - 팀원 2: 백엔드 JWT 처리 로직 검증
> - 팀원 3: DB 세션 및 토큰 저장 상태 확인
> 각자 조사 결과를 공유하고, 가장 가능성 높은 원인을 찾아줘.
```

### 3.5 에이전트 팀에서 서브에이전트 활용하기

에이전트 팀의 각 팀원은 독립적인 Claude Code 인스턴스이므로, `.claude/agents/`에 정의된 서브에이전트를 자체적으로 호출할 수 있다. 이는 "팀원이 필요할 때 외부 전문가를 불러오는 것"과 같다.

```
에이전트 팀만 사용:
  팀원 A (백엔드) ── 구현 + 리뷰 + 테스트를 모두 직접 수행

에이전트 팀 + 서브에이전트 조합:
  팀원 A (백엔드) ── 구현 직접 수행
       ├─→ [서브: code-reviewer] 자기 코드 리뷰 위임
       └─→ [서브: qa-engineer] 단위 테스트 작성 위임
```

**왜 조합하는가?**

- **컨텍스트 효율성**: 팀원도 자체 컨텍스트 윈도우 한계가 있다. 방대한 분석 작업을 서브에이전트에 위임하면 팀원의 컨텍스트를 절약할 수 있다.
- **도구 접근 제한**: 리뷰 작업을 읽기 전용 서브에이전트에 위임하면, 실수로 코드를 수정하는 일을 방지할 수 있다.
- **전문성 분리**: 팀원은 구현에 집중하고, 검증은 전문 서브에이전트에 맡기는 역할 분담이 가능하다.

**사용 예시**

{/* TODO(human): 에이전트 팀 + 서브에이전트 조합을 지시하는 프롬프트 예시를 작성하세요 */}

```
여기에 프롬프트 예시를 작성해 주세요.
```

**주의 사항**

- 서브에이전트는 팀원의 컨텍스트 안에서 순차 실행된다 (팀원 간은 병렬, 서브에이전트 호출은 순차)
- 서브에이전트가 다른 팀원에게 직접 메시지를 보낼 수는 없다. 결과는 호출한 팀원에게만 반환된다
- 과도한 서브에이전트 호출은 팀원의 작업 시간을 늘리므로, 핵심 위임만 선별해야 한다

### 3.6 알려진 제한 사항

에이전트 팀은 아직 실험적 기능이므로 다음 제한 사항을 인지해야 한다.

| 제한 사항 | 설명 |
|----------|------|
| 세션 재개 불안정 | `/resume`이 팀원에 대해 정상 동작하지 않을 수 있다 |
| 분할 패널 환경 | tmux 또는 iTerm2 필요 (VS Code 터미널, Windows Terminal 미지원) |
| 파일 잠금 없음 | 두 팀원이 같은 파일을 동시에 수정하면 마지막 쓰기가 이긴다 |
| 팀 리드 고정 | 팀 생성 세션이 영구적으로 리드이며, 변경 불가 |
| 토큰 비용 증가 | 각 팀원이 독립 컨텍스트를 가지므로, 토큰 사용량이 팀원 수에 비례 |
| 중첩 팀 불가 | 팀원이 하위 팀을 만드는 것은 불가능 |

### 3.7 에이전트 팀을 쓰면 좋은 상황

- **서로 독립적인 기능의 병렬 개발**: 백엔드/프론트엔드를 동시에 작업
- **다관점 코드 리뷰**: 보안/성능/테스트 관점을 동시에 검토
- **경쟁 가설 디버깅**: 여러 원인을 동시에 조사하여 빠르게 수렴
- **크로스 레이어 변경**: 프론트엔드, 백엔드, 테스트에 걸친 변경을 조율

### 3.8 에이전트 팀을 쓰지 않는 게 좋은 상황

- **단순한 작업**: 토큰 오버헤드가 작업 가치보다 큼
- **같은 파일을 동시에 수정해야 하는 경우**: 파일 잠금이 없어서 충돌 발생
- **순차 의존성이 강한 작업**: A 완료 후에야 B를 시작할 수 있다면, 서브에이전트가 더 적합
- **비용에 민감한 경우**: 팀원 수만큼 토큰이 배로 소모된다

## 4. Git Worktree 기반 병렬 개발

### 4.1 Git Worktree란

git worktree는 하나의 git 저장소에서 **여러 브랜치를 동시에 각각 별도의 디렉토리로 체크아웃**하는 Git 내장 기능이다.

원래는 개발자가 브랜치 전환 없이 여러 브랜치를 동시에 열어보기 위한 기능이었지만, 에이전틱 코딩 시대에 **에이전트 간 물리적 격리 수단**으로 새로운 쓰임을 찾았다.

```bash
# 원본 프로젝트 (main 브랜치)
~/project (main)

# worktree 생성: 같은 저장소, 다른 브랜치, 다른 디렉토리
git worktree add ../project-feat-auth -b feat/auth
git worktree add ../project-feat-kanban -b feat/kanban
git worktree add ../project-fix-bug -b fix/login-bug
```

결과:

```
~/project/                ← main 브랜치 (원본)
~/project-feat-auth/      ← feat/auth 브랜치 (worktree)
~/project-feat-kanban/    ← feat/kanban 브랜치 (worktree)
~/project-fix-bug/        ← fix/login-bug 브랜치 (worktree)
```

핵심 특성:
- **`.git` 데이터를 공유**한다: 저장소를 clone하는 것과 달리 디스크 공간을 거의 쓰지 않는다
- **Git 히스토리가 하나로 연결**되어 있다: worktree에서 한 커밋이 다른 worktree에서도 보인다
- **같은 브랜치를 중복 체크아웃할 수 없다**: Git이 자동으로 방지하여 혼선을 줄여준다

### 4.2 에이전틱 코딩에서 Worktree가 필요한 이유

여러 Claude Code 인스턴스를 **같은 디렉토리**에서 동시에 돌리면 문제가 발생한다.

```
문제 상황: 같은 디렉토리에서 두 에이전트 실행

에이전트 A: models.py 3번째 줄에 새 필드 추가 중...
에이전트 B: models.py 10번째 줄의 타입 수정 중...

→ 에이전트 A가 저장 → 에이전트 B가 저장
→ 에이전트 A의 변경이 덮어씌워질 수 있음
→ 양쪽 에이전트 모두 혼란에 빠짐
```

worktree는 각 에이전트에게 **물리적으로 분리된 작업 공간**을 준다. 파일 충돌이 원천적으로 차단된다.

### 4.3 "코드 마스터" 패턴 — Worktree 활용 워크플로우

실무에서 많이 사용되는 패턴이다. 여러 worktree에서 에이전트가 독립적으로 작업한 후, 코드 마스터 역할의 에이전트가 변경 사항을 확인하고 합치는 구조다.

<Mermaid>{`
flowchart TD
    TM["터미널 1: ~/project (main 브랜치)\n역할: 코드 마스터 / 오케스트레이터\nclaude: 각 브랜치의 변경 사항을 리뷰하고 main에 머지해줘"]

    WT1["터미널 2 (worktree)\nfeat/auth\nclaude: 인증 구현해줘"]
    WT2["터미널 3 (worktree)\nfeat/board\nclaude: 칸반보드 만들어줘"]
    WT3["터미널 4 (worktree)\nfix/bug\nclaude: 로그인 버그수정"]

    WT1 -->|"PR / 머지"| TM
    WT2 -->|"PR / 머지"| TM
    WT3 -->|"PR / 머지"| TM

    note["각 worktree = 물리적으로 분리된 디렉토리\n에이전트 간 파일 충돌 원천 차단"]
`}</Mermaid>

**실행 순서:**

```bash
# 1. Worktree 생성
cd ~/project
git worktree add ../project-feat-auth -b feat/auth
git worktree add ../project-feat-kanban -b feat/kanban
git worktree add ../project-fix-bug -b fix/login-bug

# 2. 각 worktree에서 Claude Code 시작 (별도 터미널)
# 터미널 2
cd ~/project-feat-auth && claude
# → "JWT 기반 인증 API를 구현해줘"

# 터미널 3
cd ~/project-feat-kanban && claude
# → "칸반 보드 UI를 만들어줘"

# 터미널 4
cd ~/project-fix-bug && claude
# → "로그인 시 토큰 만료 처리 버그를 수정해줘"

# 3. 각 에이전트가 작업 완료 후 커밋 & 푸시
# (각 터미널에서)
# → "변경 사항을 커밋하고 푸시해줘"

# 4. 코드 마스터가 리뷰 & 머지 (터미널 1)
cd ~/project
claude
# → "feat/auth, feat/kanban, fix/login-bug 브랜치의
#    변경 사항을 각각 리뷰하고, 문제없으면 main에 머지해줘.
#    충돌이 있으면 알려줘."

# 5. 정리
git worktree remove ../project-feat-auth
git worktree remove ../project-feat-kanban
git worktree remove ../project-fix-bug
```

### 4.4 Worktree 방식의 장단점

**장점:**
- **강력한 격리**: 물리적으로 다른 디렉토리이므로 파일 충돌이 불가능하다
- **안정성**: 실험적 기능에 의존하지 않는 검증된 Git 기능이다
- **자유도**: 각 세션이 완전히 독립적이므로 모델, 설정, CLAUDE.md를 다르게 쓸 수도 있다
- **비용 예측**: 일반 Claude Code 세션과 동일한 비용 구조다

**단점:**
- **수동 조율 필요**: 에이전트 간 소통이 없으므로 개발자 또는 마스터 에이전트가 조율해야 한다
- **머지 충돌 가능성**: 작업 후 브랜치를 합칠 때 Git 머지 충돌이 발생할 수 있다
- **설정 오버헤드**: worktree를 생성하고, 각 터미널에서 Claude Code를 시작하고, 종료 후 정리해야 한다
- **의존 파일 관리**: `node_modules`, `.venv` 등 버전 관리 대상이 아닌 파일은 worktree마다 별도 설치가 필요할 수 있다

### 4.5 Worktree 주요 명령어 요약

| 명령어 | 설명 |
|--------|------|
| `git worktree add <경로> -b <브랜치명>` | 새 브랜치로 worktree 생성 |
| `git worktree add <경로> <기존브랜치>` | 기존 브랜치를 worktree로 체크아웃 |
| `git worktree list` | 현재 활성 worktree 목록 확인 |
| `git worktree remove <경로>` | worktree 삭제 |
| `git worktree prune` | 더 이상 유효하지 않은 worktree 참조 정리 |

## 5. 멀티 에이전트 방식 비교 및 선택 가이드

### 5.1 전체 비교표

"멀티 에이전트"라는 용어는 여러 맥락에서 사용된다. Claude Code 생태계에서 실질적으로 사용되는 세 가지 방식을 비교한다.

| | 서브에이전트 | 에이전트 팀 | Worktree + 수동 세션 |
|---|---|---|---|
| **비유** | 외부 전문가에게 용역 | 동료들과 팀 프로젝트 | 각자 사무실에서 독립 작업 후 취합 |
| **사전 정의 필요** | ✅ `.claude/agents/`에 마크다운 파일 필수 | ❌ 자연어 프롬프트로 즉석 생성 | ❌ Git 명령어만 필요 |
| **병렬 여부** | ❌ 순차 실행 | ✅ 진정한 병렬 | ✅ 진정한 병렬 |
| **에이전트 간 소통** | ❌ 결과만 반환 | ✅ 실시간 메시지 | ❌ Git을 통해서만 |
| **조율 주체** | 메인 에이전트 (자동) | 팀 리드 에이전트 (자동) | 개발자 또는 마스터 에이전트 (수동) |
| **파일 충돌 위험** | 없음 (순차) | 있음 (같은 파일 주의) | 없음 (물리적 분리, 머지 시에만) |
| **토큰 비용** | 약간 추가 | 팀원 수 비례 증가 | 세션 수만큼 (일반 요금) |
| **설정 복잡도** | `.claude/agents/` 정의 + 프롬프트 작성 | 환경 변수 1개 활성화 | Git worktree 명령어 |
| **안정성** | 안정적 | 실험적 | 안정적 |
| **적합한 규모** | 소~중 | 중~대 | 중~대 |

### 5.2 상황별 선택 가이드

다음 의사결정 흐름을 따르면 적절한 방식을 선택할 수 있다.

```
작업이 하나의 세션에서 충분히 처리 가능한가?
│
├─ Yes → 에이전트 1개로 작업 (기본)
│
└─ No → 여러 작업을 동시에 진행해야 하는가?
         │
         ├─ No (순차 위임이면 충분)
         │    → 서브에이전트
         │
         └─ Yes (병렬 진행 필요)
              │
              ├─ 에이전트 간 실시간 소통이 필요한가?
              │    │
              │    ├─ Yes → 에이전트 팀
              │    │
              │    └─ No → 작업이 서로 다른 파일에서 이루어지는가?
              │              │
              │              ├─ Yes → Worktree + 수동 세션
              │              │
              │              └─ No (같은 파일 수정)
              │                   → 순차 처리를 권장
              │                     (서브에이전트 또는 단일 세션)
              │
              └─ 판단이 어려우면?
                   → Worktree 방식부터 시작 (가장 안정적)
```

### 5.3 실무 권장: 점진적 도입

에이전틱 코딩이 처음이라면 한 번에 복잡한 멀티 에이전트 구성을 시도하지 말고, 단계적으로 도입하는 것을 권장한다.

```
1주차: 에이전트 1개로 기본 작업에 익숙해지기
  └─ CLAUDE.md 작성, 프롬프트 기법, Plan Mode 활용

2주차: 서브에이전트로 역할 분리 시작
  └─ 코드 리뷰 서브에이전트 정의 및 활용
  └─ 테스트 작성 서브에이전트 활용

3주차: Worktree로 병렬 작업 시도
  └─ 독립적인 기능 2개를 동시에 개발
  └─ 마스터 에이전트로 리뷰 및 머지

4주차: 에이전트 팀 탐색 (선택)
  └─ 읽기 전용 작업(코드 리뷰)부터 시작
  └─ 팀 토큰 비용 모니터링
```

## 6. 실전 적용 시나리오

### 6.1 시나리오: 새 기능 추가 — 서브에이전트 활용

"태스크에 마감일 기능을 추가한다"는 작업을 서브에이전트로 수행하는 경우.

```
개발자의 지시:
─────────────

> 태스크에 due_date 필드를 추가하려고 해.
> 다음 순서로 서브에이전트를 사용해서 진행해줘:
>
> 1. backend-developer로 모델 변경 + API 수정
> 2. qa-engineer로 테스트 작성
> 3. frontend-developer로 UI 업데이트
> 4. code-reviewer로 전체 리뷰
>
> 각 단계 결과를 확인한 후 다음 단계로 진행해줘.
```

```
실행 흐름:
─────────

메인: 작업 계획 수립
  │
  ├─→ [서브: backend-developer]
  │     Task 모델에 due_date 추가
  │     Alembic 마이그레이션 생성
  │     API 스키마 업데이트
  │     ← "백엔드 변경 완료, 3개 파일 수정"
  │
  ├─→ [서브: qa-engineer]
  │     due_date 관련 테스트 12개 작성
  │     기존 테스트 회귀 확인
  │     ← "신규 테스트 12개 통과, 기존 테스트 전체 통과"
  │
  ├─→ [서브: frontend-developer]
  │     TaskCard에 마감일 표시 추가
  │     CreateTaskForm에 날짜 선택기 추가
  │     ← "프론트엔드 2개 컴포넌트 수정"
  │
  └─→ [서브: code-reviewer]
        전체 변경 사항 리뷰
        ← "Warning 1건: 마감일 유효성 검증 누락"

메인: 리뷰 결과 반영 후 커밋
```

### 6.2 시나리오: 새 기능 추가 — 에이전트 팀 활용

같은 작업을 에이전트 팀으로 수행하면 병렬로 진행된다.

```
개발자의 지시:
─────────────

> 에이전트 팀으로 태스크 마감일 기능을 구현해줘.
> - 백엔드 팀원: 모델 + API 수정
> - 프론트엔드 팀원: UI 업데이트
> - QA 팀원: 테스트 작성
> 백엔드가 스키마를 완료하면 프론트엔드에게 알려줘.
```

```
실행 흐름 (병렬):
──────────────────

시간 →

팀 리드  ██ 배정 ────────────────────────────── 종합 ██
           │
백엔드    │ [모델 변경] → [API 수정] → 완료
           │       │
           │  "스키마 완료" 메시지 ───→ │
           │                          │
프론트    │ [기존 컴포넌트 정리] ──→ [타입 업데이트 + UI] → 완료
           │
QA        │ [기존 테스트 확인] ──→ [새 테스트 작성] ──→ [실행] → 완료

         서브에이전트 대비 시간 절약 ──────────────────→
```

### 6.3 시나리오: 독립 기능 병렬 개발 — Worktree 활용

서로 무관한 두 기능을 동시에 개발하는 경우.

```bash
# 1. Worktree 생성
git worktree add ../taskflow-labels -b feat/labels
git worktree add ../taskflow-dashboard -b feat/dashboard

# 2. 터미널 2에서 (라벨 기능)
cd ~/taskflow-labels && claude
# → "태스크에 라벨 기능을 추가해줘. Label 모델을 만들고,
#    다대다 관계를 설정하고, CRUD API를 구현해줘."

# 3. 터미널 3에서 (대시보드)
cd ~/taskflow-dashboard && claude
# → "프로젝트 대시보드 페이지를 만들어줘.
#    상태별 태스크 수, 내 태스크 목록을 보여줘."

# 4. 두 작업 모두 완료 후, 터미널 1에서 (마스터)
cd ~/taskflow && claude
# → "feat/labels와 feat/dashboard 브랜치의 변경 사항을
#    각각 리뷰해줘. 문제없으면 main에 머지하고,
#    충돌이 있으면 해결해줘."

# 5. 정리
git worktree remove ../taskflow-labels
git worktree remove ../taskflow-dashboard
```
