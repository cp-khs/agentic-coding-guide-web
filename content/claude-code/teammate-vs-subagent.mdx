## 1. 기본 개념

### 팀(Team)이란?

Claude Code의 **팀(Team)** 은 여러 에이전트가 하나의 태스크 리스트를 공유하며 협업하는 구조입니다. 팀 리더가 `TeamCreate`로 팀을 생성하고, `Task` 도구로 팀원을 소환하여 작업을 분배합니다.

<Mermaid>{`
flowchart TD
    subgraph team["팀 구조"]
        TL["팀 리더 (Team Lead)"]
        MA["팀원 A (일반 팀원)"]
        MB["팀원 B (일반 팀원)"]
        MC["팀원 C (서브에이전트 팀원)"]
        TL --> MA
        TL --> MB
        TL --> MC
        note["공유: 태스크 리스트, SendMessage 통신"]
    end
`}</Mermaid>

### 일반 팀원 (Teammate)

일반 팀원은 Claude Code가 기본 제공하는 에이전트 타입(`general-purpose`, `Bash`, `Explore`, `Plan` 등)으로 소환된 팀원입니다.

- `Task` 도구의 `subagent_type` 파라미터에 빌트인 타입을 지정
- 별도의 설정 파일 없이 바로 사용 가능
- 각 빌트인 타입에 따라 사용 가능한 도구가 미리 정해져 있음

#### 빌트인 agentType 목록

| agentType | 사용 가능한 도구 | 주요 용도 |
|-----------|----------------|---------|
| `general-purpose` | 모든 도구 | 범용 작업, 파일 읽기/쓰기 모두 가능 |
| `explore` | 읽기 전용 (Read, Glob, Grep 등) | 코드베이스 탐색, 검색 전용 |
| `plan` | 읽기 전용 | 구현 계획 수립 |
| `bash` | Bash만 | 터미널 명령 실행 전용 |

### 서브에이전트 팀원 (Subagent as Teammate)

서브에이전트 팀원은 `.claude/agents/` 디렉토리에 마크다운(`.md`) 파일로 **커스텀 정의한 에이전트 타입**을 팀원으로 소환한 것입니다.

- `.claude/agents/my-agent.md` 파일에 역할, 도구 제한, 행동 규칙 등을 정의
- `Task` 도구의 `subagent_type`에 커스텀 타입명을 지정하여 소환
- 프로젝트에 맞는 특화된 역할을 부여할 수 있음

## 2. 일반 팀원과 서브에이전트 팀원의 차이

| 비교 항목 | 일반 팀원 | 서브에이전트 팀원 |
|-----------|----------|-----------------|
| **정의 위치** | Claude Code 빌트인 | `.claude/agents/*.md` 파일 |
| **설정 필요** | 없음 (바로 사용 가능) | `.md` 파일 작성 필요 |
| **역할 특화** | 범용적 (타입별 기본 역할) | 프로젝트에 맞게 세밀하게 커스텀 가능 |
| **도구 제한** | 타입별 고정 도구셋 | 마크다운 파일에서 허용/제한 도구를 직접 설정 |
| **행동 규칙** | 기본 시스템 프롬프트만 적용 | 커스텀 지시사항, 규칙, 워크플로우 정의 가능 |
| **이식성** | 어떤 프로젝트에서든 사용 가능 | 해당 `.md` 파일이 있는 프로젝트에서만 사용 가능 |
| **팀 내 통신** | SendMessage로 양방향 소통 | 동일하게 SendMessage로 양방향 소통 |
| **태스크 공유** | 팀 태스크 리스트 접근 가능 | 동일하게 팀 태스크 리스트 접근 가능 |
| **상주 여부** | 팀이 유지되는 동안 상주 | 동일하게 팀이 유지되는 동안 상주 |

### 핵심 차이: "무엇을 할 수 있는가"의 정의 방식

- **일반 팀원**: Claude Code가 미리 정해둔 역할과 도구셋을 그대로 사용
- **서브에이전트 팀원**: 개발자가 직접 역할, 도구, 행동 규칙을 마크다운으로 정의

## 3. 서브에이전트를 팀원으로 지정했을 때의 장점

### 3.1. 역할 특화 및 전문성 강화

서브에이전트 정의 파일에 구체적인 도메인 지식과 행동 규칙을 명시하면, 해당 팀원은 그 분야의 **전문가처럼 동작**합니다.

```markdown
{/* .claude/agents/db-specialist.md */}
# Database Specialist

당신은 데이터베이스 전문가입니다.
- PostgreSQL과 Prisma ORM에 특화되어 있습니다
- 마이그레이션 파일 작성 시 반드시 롤백 스크립트도 함께 작성합니다
- 인덱스 최적화를 항상 고려합니다
- 스키마 변경 시 기존 데이터 호환성을 검증합니다
```

범용 `general-purpose` 팀원에게 "DB 전문가처럼 행동해"라고 매번 지시하는 것보다 훨씬 일관적이고 신뢰할 수 있습니다.

### 3.2. 도구 접근 제한으로 안전성 확보

서브에이전트 정의에서 사용 가능한 도구를 제한하면, **실수로 위험한 작업을 수행하는 것을 방지**할 수 있습니다.

예를 들어 코드 리뷰 전용 팀원이 파일을 수정하지 못하도록 설정:

```markdown
{/* .claude/agents/reviewer.md */}
# Code Reviewer

코드 리뷰만 수행합니다. 코드를 직접 수정하지 않습니다.

## Allowed Tools
- Read
- Glob
- Grep
- Bash (git diff, git log만 허용)
```

#### 보안 관점: pane 직접 입력과 도구 제한의 관계

일반 팀원(`general-purpose`)은 tmux pane에 직접 입력된 명령도 처리하며, 팀 리더를 통한 정상 요청과 구분할 수 없습니다. 그러나 커스텀 에이전트로 도구를 제한하면, **직접 입력이 들어와도 실제로 할 수 있는 행동 자체가 제한**됩니다.

```
[일반 팀원 general-purpose]           [커스텀 에이전트 tools: Read, Grep만]
pane 직접 입력 → 모든 도구 사용 가능    pane 직접 입력 → 읽기만 가능, 파일 수정 불가
         ↓                                       ↓
    피해 범위: 무제한                       피해 범위: 제한됨
```

| 팀원 타입 | pane 직접 입력 차단 | 행동 범위 제한 |
|----------|-------------------|-------------|
| `general-purpose` | 불가 | 없음 (모든 도구) |
| 커스텀 에이전트 (`tools` 제한) | 불가 | **가능** (도구 레벨 제한) |

> **핵심**: pane 직접 입력 자체를 막을 수는 없지만, 커스텀 에이전트의 `tools` 제한으로 **피해 범위를 최소화**할 수 있습니다.

### 3.3. 프로젝트별 워크플로우 내장

팀 내에서 반복적으로 수행되는 작업 절차를 서브에이전트 정의에 내장하면, 매번 상세한 지시를 하지 않아도 됩니다.

```markdown
{/* .claude/agents/release-manager.md */}
# Release Manager

릴리스 프로세스를 관리합니다.

## 릴리스 체크리스트
1. CHANGELOG.md 업데이트
2. package.json 버전 범프
3. 모든 테스트 통과 확인
4. git tag 생성
5. 릴리스 노트 초안 작성
```

### 3.4. 팀 구성의 재사용성

한 번 정의한 서브에이전트는 **같은 프로젝트의 다른 팀에서도 재사용** 가능합니다. `.claude/agents/` 디렉토리에 파일이 존재하는 한, 어떤 팀에서든 같은 타입으로 소환할 수 있습니다.

## 4. 서브에이전트를 팀원으로 지정했을 때의 단점 및 주의사항

### 4.1. 이식성 문제: 정의 파일 의존성

**가장 큰 주의사항**입니다. 서브에이전트 팀원은 `.claude/agents/` 디렉토리에 해당 `.md` 파일이 존재해야만 소환할 수 있습니다.

```
⚠️ 소환 실패 시나리오:

프로젝트 A (.claude/agents/db-specialist.md 존재)
→ db-specialist 타입으로 팀원 소환 ✅ 성공

프로젝트 B (.claude/agents/ 에 해당 파일 없음)
→ db-specialist 타입으로 팀원 소환 ❌ 실패
```

**대응 방법:**
- 서브에이전트 정의 파일을 git으로 버전 관리하여 팀원 모두가 공유
- 공통으로 사용하는 서브에이전트는 각 프로젝트의 `.claude/agents/`에 동일하게 배치

### 4.2. 정의 파일 관리 비용

서브에이전트가 많아지면 `.claude/agents/` 디렉토리의 `.md` 파일들을 **유지보수**해야 합니다.

- 프로젝트 구조가 변경되면 에이전트 정의도 업데이트 필요
- 정의가 너무 제한적이면 예상치 못한 상황에서 대응력이 떨어짐
- 정의가 너무 느슨하면 서브에이전트를 쓰는 의미가 퇴색

### 4.3. 디버깅의 어려움

서브에이전트 팀원이 예상대로 동작하지 않을 때, 문제가 **정의 파일의 지시사항 때문인지, 모델의 일반적 한계 때문인지** 구분하기 어려울 수 있습니다.

### 4.4. 도구 제한의 양날의 검

도구를 과도하게 제한하면 팀원이 필요한 작업을 수행하지 못할 수 있습니다. 예를 들어, 코드 리뷰어에게 `Bash`를 완전히 차단하면 테스트 실행 결과를 확인하지 못합니다.

## 5. 선택 가이드: 언제 무엇을 쓸까?

### 일반 팀원을 선택하는 경우

| 상황 | 이유 |
|------|------|
| **빠르게 팀을 구성해야 할 때** | 별도 설정 없이 바로 소환 가능 |
| **범용적인 작업을 수행할 때** | 코딩, 탐색, 실행 등 일반적 작업 |
| **일회성 팀 구성일 때** | 정의 파일을 만들 필요가 없음 |
| **여러 프로젝트에서 동일하게 사용할 때** | 프로젝트 의존성 없음 |
| **팀원 역할이 단순할 때** | "코드 작성", "테스트 실행" 등 명확한 단일 역할 |

### 서브에이전트 팀원을 선택하는 경우

| 상황 | 이유 |
|------|------|
| **프로젝트 고유의 규칙이 있을 때** | 코딩 컨벤션, 아키텍처 규칙 등을 내장 |
| **안전성이 중요한 작업일 때** | 도구 제한으로 위험한 작업 방지 |
| **반복적인 팀 구성이 필요할 때** | 한 번 정의하면 계속 재사용 |
| **도메인 전문성이 필요할 때** | DB, 보안, 인프라 등 특화 역할 |
| **복잡한 워크플로우가 있을 때** | 체크리스트, 절차를 에이전트에 내장 |
| **팀원의 행동 범위를 통제해야 할 때** | 읽기 전용, 특정 디렉토리만 접근 등 |

### 의사결정 플로우차트

```
팀원을 추가해야 한다
│
├─ 특별한 역할 규칙이나 도구 제한이 필요한가?
│  ├─ YES → .claude/agents/에 정의 파일이 있는가?
│  │         ├─ YES → 서브에이전트 팀원으로 소환
│  │         └─ NO  → 정의 파일을 먼저 작성 → 서브에이전트 팀원으로 소환
│  │
│  └─ NO  → 일반 팀원으로 소환
│
└─ 빠르게 임시 팀원이 필요한가?
   └─ YES → 일반 팀원으로 소환
```

## 6. 실제 사용 예시

### 예시 1: 일반 팀원으로 팀 구성

간단한 기능 개발 팀을 빠르게 구성하는 경우입니다.

```python
# 팀 리더의 작업 흐름 (의사 코드)

# 1. 팀 생성
TeamCreate(team_name="feature-auth")

# 2. 일반 팀원 소환
Task(
    name="coder",
    subagent_type="general-purpose",  # 빌트인 타입
    team_name="feature-auth",
    prompt="인증 모듈의 코드를 작성하는 팀원입니다..."
)

Task(
    name="tester",
    subagent_type="general-purpose",  # 빌트인 타입
    team_name="feature-auth",
    prompt="테스트를 작성하고 실행하는 팀원입니다..."
)

# 3. 태스크 할당 및 작업 진행
TaskCreate(subject="로그인 API 구현", ...)
TaskUpdate(taskId="1", owner="coder")
```

### 예시 2: 서브에이전트를 팀원으로 구성

프로젝트 규칙이 명확하고, 역할별 도구 제한이 필요한 경우입니다.

**먼저 서브에이전트를 정의합니다:**

```markdown
{/* .claude/agents/frontend-dev.md */}
# Frontend Developer

React + TypeScript 프론트엔드 개발 전문가입니다.

## 규칙
- 컴포넌트는 반드시 함수형 컴포넌트로 작성합니다
- 상태 관리는 Zustand를 사용합니다
- 스타일링은 Tailwind CSS만 사용합니다
- src/components/ 하위에만 파일을 생성합니다

## Allowed Tools
- Read, Write, Edit, Glob, Grep, Bash
```

```markdown
{/* .claude/agents/qa-engineer.md */}
# QA Engineer

품질 보증 전문가입니다. 코드를 직접 수정하지 않고, 문제를 발견하여 보고합니다.

## 규칙
- 코드를 읽고 분석만 합니다
- 발견한 버그/이슈는 태스크로 생성합니다
- 테스트는 실행하되 테스트 파일은 수정하지 않습니다

## Allowed Tools
- Read, Glob, Grep, Bash
```

**그런 다음 팀에서 소환합니다:**

```python
# 팀 리더의 작업 흐름 (의사 코드)

# 1. 팀 생성
TeamCreate(team_name="sprint-42")

# 2. 서브에이전트 팀원 소환
Task(
    name="fe-dev",
    subagent_type="frontend-dev",  # 커스텀 타입 (.claude/agents/frontend-dev.md)
    team_name="sprint-42",
    prompt="프론트엔드 개발을 담당합니다..."
)

Task(
    name="qa",
    subagent_type="qa-engineer",   # 커스텀 타입 (.claude/agents/qa-engineer.md)
    team_name="sprint-42",
    prompt="QA를 담당합니다..."
)

# 3. 태스크 할당 (TaskCreate로 생성 후, TaskUpdate로 담당자 지정)
TaskCreate(subject="사용자 프로필 페이지 구현", description="...")
TaskUpdate(taskId="1", owner="fe-dev")
TaskCreate(subject="기존 인증 플로우 QA 검증", description="...")
TaskUpdate(taskId="2", owner="qa")
```

### 예시 3: 혼합 구성 (일반 + 서브에이전트)

실전에서는 **일반 팀원과 서브에이전트 팀원을 함께 사용**하는 경우가 많습니다.

```python
TeamCreate(team_name="release-v2")

# 서브에이전트 팀원: 프로젝트 규칙을 아는 전문가
Task(
    name="releaser",
    subagent_type="release-manager",  # 커스텀: 릴리스 프로세스 내장
    team_name="release-v2",
    prompt="릴리스 프로세스를 관리합니다..."
)

# 일반 팀원: 범용 작업 수행
Task(
    name="fixer",
    subagent_type="general-purpose",  # 빌트인: 범용 코드 수정
    team_name="release-v2",
    prompt="릴리스 차단 버그를 수정합니다..."
)

# 일반 팀원: 문서 작성
Task(
    name="doc-writer",
    subagent_type="general-purpose",  # 빌트인: 범용 문서 작성
    team_name="release-v2",
    prompt="릴리스 노트와 문서를 작성합니다..."
)
```

## 7. 팀원 라이프사이클 관리

팀원의 소환부터 종료, 재소환까지의 전체 흐름을 이해하면 팀 운영이 훨씬 수월해집니다.

### 7.1. 팀원 소환 방법

팀원 소환은 **사용자 → 팀 리더 → 팀원** 순서로 이루어집니다.

```
사용자: "tester 소환해줘"
    ↓
팀 리더 (Claude Code): Task 도구 호출
    ↓
팀원 프로세스 생성 + 팀 config.json에 등록
    ↓
팀원 활동 시작
```

**소환 시 핵심 포인트:**

- **사용자는 자연어로 요청**하면 됩니다. "tester 추가해줘", "DB 전문가 팀원 소환해줘" 등
- 팀 리더가 내부적으로 `Task` 도구를 호출하며, `team_name`을 지정하면 해당 팀에 팀원으로 등록됩니다
- `subagent_type`으로 **빌트인 타입**(`general-purpose`, `Bash`, `Explore` 등) 또는 **커스텀 타입**(`.claude/agents/` 디렉토리의 `.md` 파일명)을 지정할 수 있습니다

```python
# 빌트인 타입으로 소환
Task(name="tester", subagent_type="general-purpose", team_name="my-team", ...)

# 커스텀 타입으로 소환 (.claude/agents/qa-engineer.md 필요)
Task(name="tester", subagent_type="qa-engineer", team_name="my-team", ...)
```

소환된 팀원은 `~/.claude/teams/{team-name}/config.json`의 `members` 배열에 자동 등록됩니다.

### 7.2. 정상 종료: `shutdown_request` (권장)

팀원을 종료하는 **가장 안전하고 권장되는 방법**입니다.

소환(7.1)과 마찬가지로, 사용자는 **자연어로 팀 리더에게 종료를 요청**하면 됩니다. 팀 리더가 내부적으로 `shutdown_request`를 처리합니다.

**사용자 요청 예시:**
- "tester 종료해줘"
- "doc-writer 셧다운해줘"
- "tester 꺼줘"

**전체 흐름:**

```
사용자: "tester 종료해줘" (자연어 요청)
    ↓
팀 리더: SendMessage(type: "shutdown_request", recipient: "tester")
    ↓
팀원: shutdown_request 수신
    ↓
팀원: SendMessage(type: "shutdown_response", approve: true)
    ↓
팀원 프로세스 종료 + config.json에서 자동 정리
```

> 사용자가 직접 도구를 호출하거나 명령어를 입력할 필요가 없습니다. 자연어 요청만으로 충분합니다.

**정상 종료의 장점:**

| 항목 | 상태 |
|------|------|
| 팀원 프로세스 | 정상 종료됨 |
| config.json | 자동으로 정리됨 |
| mailbox 메시지 | 깔끔하게 처리됨 |
| 같은 이름으로 재소환 | 이름 충돌 없이 가능 |

> **팁**: 팀 작업이 끝나면 항상 `shutdown_request`로 팀원을 정리한 후 `TeamDelete`로 팀을 해산하는 것이 깔끔합니다.

### 7.3. 강제 종료: `/exit` (비권장)

tmux 패널에서 직접 Claude Code의 `/exit` 명령을 실행하여 종료하는 방법입니다. **팀 시스템을 우회하는 종료**이므로 여러 부작용이 있습니다.

```
사용자: tmux pane에서 직접 /exit 입력
    ↓
Claude Code 프로세스만 종료
    ↓
tmux pane은 shell 상태로 남음
    ↓
⚠️ config.json에 좀비 항목이 남음
⚠️ 팀 시스템이 종료를 인식하지 못함
```

**`/exit` 강제 종료 시 발생하는 문제:**

| 항목 | 상태 |
|------|------|
| Claude Code 프로세스 | 종료됨 |
| tmux pane | shell 상태로 남음 (좀비 pane) |
| config.json | **좀비 항목이 남음** (팀이 종료를 인식 못함) |
| mailbox 메시지 | 쌓인 메시지는 재소환 후 그대로 수신됨 |
| 같은 이름으로 재소환 | **이름 충돌 발생 가능** |

### 7.4. `/exit` 후 재소환 방법

강제 종료 후 팀원을 다시 소환해야 할 때, 두 가지 방법이 있습니다.

#### 방법 A: 수동 정리 후 재소환 (깔끔한 방법)

config.json에서 기존 좀비 항목을 직접 제거한 후 재소환합니다.

```bash
# 1. config.json 확인
cat ~/.claude/teams/my-team/config.json
```

```json
{
  "members": [
    {
      "name": "team-lead",
      "agentId": "abc-123",
      "agentType": "general-purpose"
    },
    {
      "name": "tester",        ← 이 좀비 항목을 제거
      "agentId": "def-456",
      "agentType": "general-purpose"
    }
  ]
}
```

```bash
# 2. 좀비 항목 제거 후 저장
# ⚠️ 주의: trailing comma를 남기지 않도록 주의!
```

```json
{
  "members": [
    {
      "name": "team-lead",
      "agentId": "abc-123",
      "agentType": "general-purpose"
    }
  ]
}
```

```bash
# 3. 팀 리더에게 재소환 요청
# → "tester를 다시 소환해줘"
# → 기존과 동일한 이름으로 깔끔하게 소환됨 ✅
```

> **주의**: JSON 파일 편집 시 **trailing comma**(마지막 항목 뒤의 쉼표)가 남으면 JSON 파싱 오류가 발생합니다. 반드시 올바른 JSON 형식을 유지하세요.

#### 방법 B: 그냥 재소환 (빠른 방법)

config.json을 수정하지 않고, 팀 리더에게 바로 재소환을 요청합니다.

```bash
# 1. 팀 리더에게 재소환 요청
# → "tester를 다시 소환해줘"

# 2. 이름 충돌로 자동으로 "tester-2"로 생성됨
# config.json:
{
  "members": [
    { "name": "team-lead", ... },
    { "name": "tester", ... },      ← 좀비 (비활성)
    { "name": "tester-2", ... }     ← 새로 소환된 팀원
  ]
}

# 3. 필요 시 config.json에서 이름 수정 가능
# "tester-2" → "tester" (좀비 항목 제거 후)
```

#### 방법 비교

| 항목 | 방법 A (수동 정리) | 방법 B (그냥 재소환) |
|------|-------------------|---------------------|
| 소요 시간 | 느림 (JSON 편집 필요) | 빠름 (바로 요청) |
| 이름 유지 | 기존 이름 유지 가능 | `tester-2`로 변경됨 |
| config.json 상태 | 깔끔함 | 좀비 항목 잔존 |
| JSON 오류 위험 | trailing comma 주의 | 없음 |
| 권장 상황 | 장기 운영 팀 | 임시/일회성 팀 |

### 7.5. 주의사항 정리

#### config.json 직접 수정 시 주의

```json
// ❌ 잘못된 예: trailing comma
{
  "members": [
    { "name": "team-lead", ... },   ← 여기에 쉼표가 남으면 오류!
  ]
}

// ✅ 올바른 예
{
  "members": [
    { "name": "team-lead", ... }
  ]
}
```

#### 프로세스 생성과 팀 등록은 별개 단계

팀원 소환 과정은 내부적으로 여러 단계를 거칩니다:

```
Task 도구 호출
  ├── 1단계: 에이전트 프로세스 생성 (subprocess spawn)
  ├── 2단계: Claude Code 인스턴스 초기화
  └── 3단계: 팀 config.json에 등록  ← 이 시점에서 팀원으로 인식
```

중간 단계에서 오류가 발생하면:
- **1단계 성공 + 2~3단계 실패**: 프로세스만 남는 **좀비 프로세스** 발생
- 이 경우 해당 프로세스를 수동으로 정리해야 합니다

> **tmux 환경의 경우**: tmux에서 팀원을 실행 중이라면, 좀비 프로세스가 tmux pane에 남을 수 있습니다. 이때는 아래 명령으로 정리합니다.

```bash
# 좀비 pane 확인 (tmux 환경에서만 해당)
tmux list-panes -a

# 특정 pane 종료
tmux kill-pane -t {pane-id}
```

### 7.6. 라이프사이클 전체 흐름도

<Mermaid>{`

    A["소환 (Spawn)"] -->|"Task 도구\nteam_name 지정"| B["활동중 (Active)"]

    B --> D["강제 종료\n/exit"]

    D --> F["좀비 항목 잔존\n수동 정리 필요"]

| 재사용성 | **매번 프롬프트 작성** | **한 번 정의, 반복 사용** |
