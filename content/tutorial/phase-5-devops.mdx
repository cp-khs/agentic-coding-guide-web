# Phase 5: DevOps 및 QA (30분)

## 6.1 Docker 환경 최적화

devops-engineer 서브에이전트를 사용해서 다음을 수행해줘:

**중요: 다음 디렉토리 구조를 정확히 따라줘:**

### Dockerfile 위치 및 구조

1. **백엔드 Dockerfile** (docker 디렉토리에 생성):
   - `docker/backend.Dockerfile` (프로덕션용 멀티스테이지)
   - `docker/backend.dev.Dockerfile` (개발용)
   - 멀티스테이지 빌드 (builder, production 스테이지)
   - 비루트 사용자 설정 (appuser, UID 1001)
   - .dockerignore 설정 (backend/.dockerignore)

2. **프론트엔드 Dockerfile** (docker 디렉토리에 생성):
   - `docker/frontend.Dockerfile` (프로덕션용 멀티스테이지)
   - `docker/frontend.dev.Dockerfile` (개발용)
   - Next.js standalone 출력 모드
   - 멀티스테이지 빌드 (deps, builder, development, production 스테이지)
   - 비루트 사용자 설정 (nextjs, UID 1001)
   - .dockerignore 설정 (frontend/.dockerignore)

3. **docker-compose.yml 업데이트**:
   - 개발 환경용 설정
   - Dockerfile 경로 명시:
     ```yaml
     backend:
       build:
         context: .
         dockerfile: docker/backend.dev.Dockerfile

     frontend:
       build:
         context: .
         dockerfile: docker/frontend.dev.Dockerfile
         target: development  # 멀티스테이지 사용 시
     ```
   - **Named volume 사용** (익명 볼륨 사용 금지):
     ```yaml
     volumes:
       - backend_cache:/app/.cache
       - frontend_node_modules:/app/node_modules
       - frontend_next:/app/.next

     volumes:
       postgres_data:
       backend_cache:
       frontend_node_modules:
       frontend_next:
     ```
   - 헬스체크 추가 (각 서비스)
   - 환경 변수 관리 (.env 파일 활용)

## 6.2 CI/CD 파이프라인

GitHub Actions 워크플로우 파일을 만들어줘.

.github/workflows/ci.yml:
- PR 생성 시 트리거
- 백엔드 린트 (ruff) + 테스트 (pytest)
- 프론트엔드 린트 (eslint) + 타입 체크 + 빌드
- PostgreSQL 서비스 컨테이너 사용

## 6.3 최종 통합 테스트

qa-engineer 서브에이전트를 사용해서 전체 통합 테스트를 수행해줘.

1. Docker Compose로 전체 서비스를 올려서 E2E 테스트
2. 주요 사용자 시나리오 테스트:
   - 회원가입 → 로그인 → 프로젝트 생성 → 태스크 생성
     → 상태 변경 → 댓글 작성
3. 에러 케이스 테스트:
   - 인증 없이 API 접근
   - 권한 없는 프로젝트/태스크 접근
   - 잘못된 입력값
4. 테스트 결과 보고서 생성
