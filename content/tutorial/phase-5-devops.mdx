## 1.1 Docker 환경 최적화

```
devops-engineer 서브에이전트를 사용해서 다음을 수행해줘:

**중요: 다음 디렉토리 구조를 정확히 따라줘:**
```

### Dockerfile 위치 및 구조

1. **백엔드 Dockerfile** (docker 디렉토리에 생성):
   - `docker/backend.Dockerfile` (프로덕션용 멀티스테이지)
   - `docker/backend.dev.Dockerfile` (개발용)
   - 멀티스테이지 빌드 (builder, production 스테이지)
   - 비루트 사용자 설정 (appuser, UID 1001)
   - .dockerignore 설정 (backend/.dockerignore)

2. **프론트엔드 Dockerfile** (docker 디렉토리에 생성):
   - `docker/frontend.Dockerfile` (프로덕션용 멀티스테이지)
   - `docker/frontend.dev.Dockerfile` (개발용)
   - Next.js standalone 출력 모드
   - 멀티스테이지 빌드 (deps, builder, development, production 스테이지)
   - 비루트 사용자 설정 (nextjs, UID 1001)
   - .dockerignore 설정 (frontend/.dockerignore)

3. **docker-compose.yml 업데이트**:
   - 개발 환경용 설정
   - Dockerfile 경로 명시:
     ```yaml
     backend:
       build:
         context: .
         dockerfile: docker/backend.dev.Dockerfile

     frontend:
       build:
         context: .
         dockerfile: docker/frontend.dev.Dockerfile
         target: development  # 멀티스테이지 사용 시
     ```
   - **Named volume 사용** (익명 볼륨 사용 금지):
     ```yaml
     volumes:
       - backend_cache:/app/.cache
       - frontend_node_modules:/app/node_modules
       - frontend_next:/app/.next

     volumes:
       postgres_data:
       backend_cache:
       frontend_node_modules:
       frontend_next:
     ```
   - 헬스체크 추가 (각 서비스)
   - 환경 변수 관리 (.env 파일 활용)

## 1.3 최종 통합 테스트

```
qa-engineer 서브에이전트를 사용해서 전체 통합 테스트를 수행해줘.

1. Docker Compose로 전체 서비스를 올려서 E2E 테스트
2. 주요 사용자 시나리오 테스트:
   - 회원가입 → 로그인 → 프로젝트 생성 → 태스크 생성
     → 상태 변경 → 댓글 작성
3. 에러 케이스 테스트:
   - 인증 없이 API 접근
   - 권한 없는 프로젝트/태스크 접근
   - 잘못된 입력값
4. 테스트 결과 보고서 생성
```

### 🔍 완료 확인 방법

**방법 1: Docker 최적화 확인**
```
"Dockerfile이 최적화되었는지 확인해줘.
- 멀티스테이지 빌드가 적용되었나?
- .dockerignore 파일이 있나?
- 비루트 사용자로 실행되나?
- 헬스체크가 설정되었나?"
```

**방법 2: CI/CD 파이프라인 확인**
```
"GitHub Actions 워크플로우 파일을 보여줘.
어떤 단계들이 포함되어 있고, 언제 실행되는가?"
```

또는 직접 확인:
```bash
# CI/CD 워크플로우 파일 확인
cat .github/workflows/ci.yml

# Docker 파일 확인
ls -la docker/ backend/Dockerfile frontend/Dockerfile
cat docker-compose.yml
```

**방법 3: 전체 서비스 실행 테스트**
```
"Docker Compose로 전체 서비스를 실행해줘.
그리고 각 서비스가 정상적으로 기동되는지 확인해줘."
```

직접 실행:
```bash
# 전체 서비스 시작
docker-compose up -d

# 서비스 상태 확인
docker-compose ps

# 로그 확인
docker-compose logs -f

# 헬스체크 확인
docker-compose exec backend curl http://localhost:8000/health
docker-compose exec frontend curl http://localhost:3000

# 서비스 접속 테스트
# 백엔드: http://localhost:8000/docs
# 프론트엔드: http://localhost:3000
```

**방법 4: 통합 테스트 확인**
```
"전체 통합 테스트를 실행해줘.
주요 사용자 시나리오(회원가입 → 로그인 → 프로젝트 생성 → 태스크 관리)가
모두 통과하는지 확인해줘."
```

**방법 5: 프로덕션 준비 체크리스트**
```
"프로덕션 배포를 위한 체크리스트를 만들어줘.
빠진 부분이나 개선이 필요한 부분이 있는지 검토해줘."
```

**다음 단계로 진행해도 되는 경우:**
- ✅ Docker Compose로 전체 스택이 정상 실행됨
- ✅ 백엔드 API가 정상 응답함 (http://localhost:8000/docs 접속 가능)
- ✅ 프론트엔드가 정상 렌더링됨 (http://localhost:3000 접속 가능)
- ✅ 주요 기능(회원가입, 로그인, 프로젝트/태스크 관리)이 E2E로 작동함
- ✅ CI/CD 워크플로우 파일이 있고 문법 오류가 없음
- ✅ Docker 이미지 빌드가 성공함

**최종 확인:**
```bash
# 전체 빌드 테스트

```
