# Phase 4: 멀티 에이전트 협업 실습 (60분)

**이 Phase가 이 튜토리얼의 핵심이다.** 앞서 정의한 서브에이전트와 에이전트 팀을 활용하여 여러 에이전트가 협업하는 워크플로우를 실습한다.

## 1.1 시나리오: "새로운 기능 추가"를 멀티 에이전트로 수행하기

다음 기능을 추가한다고 가정하자: **"태스크에 마감일(due_date)과 라벨(labels) 기능 추가"**

이 기능은 백엔드 스키마 변경, API 수정, 프론트엔드 UI 업데이트, 테스트 작성이 모두 필요하므로 멀티 에이전트 협업의 좋은 예시다.

## 1.2 방법 1: 서브에이전트를 활용한 순차 협업

하나의 메인 세션에서 서브에이전트에게 작업을 위임하는 방식이다.

```
태스크에 두 가지 새 기능을 추가하려고 해:
1. 마감일 (due_date: datetime, nullable)
2. 라벨 (labels: 다대다 관계, Label 모델 새로 생성)

다음 순서로 서브에이전트를 사용해서 진행해줘:

1. backend-developer 서브에이전트로:
   - Task 모델에 due_date 필드 추가
   - Label 모델과 task_labels 연결 테이블 생성
   - Alembic 마이그레이션 생성 및 실행
   - 관련 Pydantic 스키마 업데이트
   - API 엔드포인트 수정/추가 (라벨 CRUD, 태스크에 라벨 연결)

2. qa-engineer 서브에이전트로:
   - 새 기능에 대한 테스트 코드 작성
   - 기존 테스트가 깨지지 않는지 확인

3. frontend-developer 서브에이전트로:
   - TaskCard에 마감일 표시 (기한 임박 시 색상 변경)
   - 라벨 뱃지 표시
   - CreateTaskForm에 마감일 선택기와 라벨 선택 추가
   - TaskDetailModal에 마감일/라벨 편집 기능 추가

4. code-reviewer 서브에이전트로:
   - 전체 변경 사항에 대한 코드 리뷰 실행
   - 보안, 성능, 코드 품질 관점으로 검토

각 단계의 결과를 보여주고, 다음 단계로 넘어가기 전에 확인해줘.
```

**이 방식의 장점**: 순서가 보장되고, 각 단계의 결과를 확인할 수 있다. 의존 관계가 있는 작업(예: 백엔드 완성 후 프론트엔드 작업)에 적합하다.

## 1.3 방법 2: 에이전트 팀을 활용한 병렬 협업

에이전트 팀을 사용하면 여러 작업을 동시에 진행할 수 있다.

먼저 에이전트 팀 기능이 활성화되어 있는지 확인한다.

```bash
echo $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS
# 출력: 1 이어야 함
```

그런 다음 Claude Code에서 팀을 구성한다.

```
에이전트 팀을 만들어서 다음 작업을 병렬로 진행해줘:

팀 구성:
- 백엔드 개발자: Task 모델에 due_date 추가, Label 모델 생성,
  API 엔드포인트 수정
- 프론트엔드 개발자: 현재 API 스펙을 기반으로 UI 컴포넌트 업데이트
  (마감일 표시, 라벨 뱃지)
- QA 엔지니어: 기존 테스트 확인 및 새 기능 테스트 작성

백엔드 개발자가 스키마 변경을 완료하면 프론트엔드 개발자에게
알려서 타입을 업데이트하도록 해줘.
QA 엔지니어는 각 개발자의 작업이 끝날 때마다 테스트를 실행해줘.
```

에이전트 팀이 실행되는 흐름은 다음과 같다.

<Mermaid>{`
sequenceDiagram
  participant L as 팀 리드
  participant B as 백엔드
  participant F as 프론트엔드
  participant Q as QA

  L->>B: 작업 배정 (모델 변경 + API)
  L->>F: 작업 배정 (기존 UI 정리)
  L->>Q: 작업 배정 (기존 테스트 확인)

  par 병렬 실행
    B->>B: 모델 변경
  and
    F->>F: 기존 UI 정리
  and
    Q->>Q: 기존 테스트 확인
  end

  B->>F: 스키마 완료 메시지
  B->>B: API 수정

  par 병렬 실행
    F->>F: 타입 업데이트
  and
    Q->>Q: 새 테스트 작성
  end

  B->>L: 완료 보고
  F->>L: 완료 보고
  Q->>L: 완료 보고
  L->>L: 결과 종합 → 리뷰
`}</Mermaid>

**이 방식의 장점**: 독립적으로 진행 가능한 작업을 병렬로 수행하여 시간을 절약한다. 팀원 간 메시지 교환으로 의존 관계도 처리할 수 있다.

**주의 사항**: 에이전트 팀은 토큰 사용량이 팀원 수에 비례하여 증가한다. 단순한 작업에는 오버헤드가 크므로, 병렬 탐색이 실질적인 가치를 제공하는 경우에만 사용하라.

## 1.4 방법 3: 하이브리드 워크플로우 (권장)

실무에서는 서브에이전트와 에이전트 팀을 상황에 따라 조합하는 것이 가장 효과적이다.

<Mermaid>{`
flowchart TD
  A["1단계: 설계 - 메인 세션\nPlan Mode로 기능 설계 및 작업 분해"]
  B["2단계: 구현 - 에이전트 팀\n백엔드 / 프론트엔드 / 테스트 병렬 진행"]
  C["3단계: 검증 - 서브에이전트\ncode-reviewer 서브에이전트로 리뷰"]
  D["4단계: 수정 - 메인 세션\n리뷰 결과 반영, 최종 테스트"]
  E["5단계: 배포 - 서브에이전트\ndevops-engineer 서브에이전트로 배포 설정"]

  A --> B --> C --> D --> E
`}</Mermaid>

이를 실제로 실행해보자.

**1단계: 설계 (메인 세션, Plan Mode)**

```
[Plan Mode] 태스크에 마감일과 라벨 기능을 추가하려고 해.
백엔드, 프론트엔드, 테스트 각각에서 필요한 변경 사항을 정리하고,
작업 간 의존 관계를 분석해줘.
어떤 작업을 병렬로 진행할 수 있는지도 알려줘.
```

**2단계: 병렬 구현 (에이전트 팀)**

```
에이전트 팀을 만들어서 방금 계획한 대로 구현을 진행해줘.

- 백엔드 팀원: 모델 + 마이그레이션 + API 수정
- 프론트엔드 팀원: 컴포넌트 업데이트 + 타입 정의
- QA 팀원: 테스트 작성 + 실행

각 팀원은 작업 완료 시 팀 리드에게 보고해줘.
```

**3단계: 코드 리뷰 (서브에이전트)**

```
code-reviewer 서브에이전트를 사용해서 이번에 변경된 모든 파일을
리뷰해줘. 보안, 성능, 코드 품질 관점에서 검토하고,
심각도별로 이슈를 정리해줘.
```

**4단계: 수정 및 최종 확인 (메인 세션)**

```
리뷰에서 나온 이슈들을 수정해줘.
수정 후 전체 테스트를 다시 실행해서 모두 통과하는지 확인해줘.
```

**5단계: 커밋 및 정리**

```
변경 사항을 의미 있는 커밋 단위로 나눠서 커밋해줘.
커밋 메시지는 Conventional Commits 형식으로.
```

## 1.5 실습: PR 리뷰를 에이전트 팀으로 수행하기

에이전트 팀의 가장 즉각적인 가치를 느낄 수 있는 사례가 코드 리뷰다.

```
에이전트 팀으로 현재까지의 전체 코드를 리뷰해줘.
세 명의 리뷰어를 각각 다른 관점에서 투입해줘:

- 보안 전문가: SQL 인젝션, XSS, 인증 우회, 민감 데이터 노출 검토
- 성능 전문가: N+1 쿼리, 불필요한 연산, DB 인덱스, 캐싱 필요 여부
- 테스트 전문가: 테스트 커버리지, 엣지 케이스, 에러 시나리오 확인

각 리뷰어의 결과를 종합해서 우선순위별로 정리해줘.
```

이 실습을 통해 에이전트 팀이 어떻게 서로 독립적으로 작업하면서도 종합적인 결과를 만들어내는지 경험할 수 있다.

**✅ Phase 4 체크포인트**

- [ ] Label 모델 생성 및 다대다 관계 설정 (백엔드)
- [ ] 마이그레이션 생성 및 적용
- [ ] API 엔드포인트 업데이트 (라벨 CRUD)
- [ ] 프론트엔드 UI에 마감일/라벨 표시 추가
- [ ] 테스트 코드 작성 및 통과
- [ ] 코드 리뷰 완료

### 🔍 완료 확인 방법

**방법 1: 변경 사항 확인**
```
"Phase 4에서 추가한 기능이 제대로 구현되었는지 확인해줘.
- Task 모델에 due_date 필드가 추가되었나?
- Label 모델이 생성되고 Task와 연결되었나?
- 마이그레이션이 적용되었나?
- 프론트엔드에 마감일과 라벨이 표시되나?"
```

**방법 2: Git 변경 이력 확인**
```
"Phase 4에서 변경된 파일 목록을 보여줘.
어떤 파일들이 수정되었고, 주요 변경 내용은 무엇인가?"
```

또는 직접 명령 실행:
```bash
# 최근 커밋 확인
git log --oneline -5

# 변경된 파일 목록
git diff --name-only HEAD~3 HEAD

# 특정 파일의 변경 내용 확인
git diff HEAD~3 HEAD backend/app/models/task.py
```

**방법 3: 기능 테스트**
```
"새로 추가한 기능을 테스트해보고 싶어.
1. 백엔드 테스트가 모두 통과하는지 확인해줘
2. API 문서(Swagger)에 라벨 관련 엔드포인트가 추가되었는지 보여줘
3. 프론트엔드에서 태스크를 생성할 때 마감일을 설정할 수 있는지 확인해줘"
```

직접 테스트:
```bash
# 백엔드 테스트 실행
cd backend
pytest

# API 문서 확인
# http://localhost:8000/docs 접속

# 프론트엔드 확인
# http://localhost:3000 접속하여 태스크 생성 화면 확인
```

**방법 4: 에이전트 협업 결과 확인**
```
"이번 Phase에서 서브에이전트들이 어떤 작업을 했는지 요약해줘.
각 서브에이전트(backend-developer, frontend-developer, qa-engineer, code-reviewer)의
기여 내용을 정리해줘."
```

**다음 Phase로 진행해도 되는 경우:**
- ✅ 백엔드에 due_date와 Label 모델이 추가됨
- ✅ 마이그레이션이 성공적으로 적용됨
- ✅ API 엔드포인트가 정상 작동함 (Swagger에서 확인)
- ✅ 프론트엔드 UI에 새 기능이 반영됨
- ✅ 모든 테스트가 통과함
