## 1. 튜토리얼 프로젝트 소개: TaskFlow

### 1.1 프로젝트 개요

**TaskFlow**는 팀 단위 태스크 관리 웹 애플리케이션이다. 칸반 보드 스타일의 UI로 태스크를 관리하며, 팀원 배정, 상태 추적, 댓글 기능을 제공한다.

이 프로젝트는 에이전틱 코딩을 학습하기에 적합한 규모와 복잡도를 갖추고 있다. 레거시 방식으로는 백엔드·프론트엔드·DevOps·QA 각 1명이 필요한 규모지만, Claude Code를 활용하면 1명의 개발자가 약 4시간 내에 완성할 수 있다.

### 1.2 기술 스택

| 영역 | 기술 |
|------|------|
| 백엔드 | Python 3.12, FastAPI, SQLAlchemy 2.0, Alembic |
| 프론트엔드 | Next.js 15 (App Router), TypeScript, Tailwind CSS |
| 데이터베이스 | PostgreSQL 16 |
| 인증 | JWT (python-jose, passlib) |
| 인프라 | Docker, Docker Compose |
| 테스트 | pytest (백엔드), Vitest (프론트엔드) |

## 2. Phase 1: 프로젝트 구성

### 2.1 프로젝트 디렉토리 생성 및 Claude Code 시작

```bash
# 프로젝트 디렉토리 생성
mkdir taskflow && cd taskflow
git init

# Claude Code 시작
claude
```

### 2.2 첫 번째 지시: CLAUDE.md 생성

Claude Code를 시작하면 가장 먼저 CLAUDE.md를 작성하도록 지시한다. 에이전트에게 프로젝트의 전체 그림을 알려주는 것이 모든 작업의 기초가 된다.

```
다음 프로젝트를 위한 CLAUDE.md 파일을 생성해줘.

프로젝트명: TaskFlow (팀 태스크 관리 웹앱)
기술 스택:
- 백엔드: Python 3.12, FastAPI, SQLAlchemy 2.0, Alembic, JWT 인증
- 프론트엔드: Next.js 15 (App Router), TypeScript, Tailwind CSS
- DB: PostgreSQL 16
- 인프라: Docker, Docker Compose

주요 기능: 사용자 인증, 프로젝트 관리, 태스크 CRUD (칸반 보드), 담당자 배정, 댓글, 대시보드

코딩 컨벤션:
- 백엔드: PEP 8, type hint 필수, async/await 패턴
- 프론트엔드: ESLint + Prettier, 함수형 컴포넌트, Server Components 우선
- 커밋: Conventional Commits
- 변경 전 반드시 git branch 생성
- API 엔드포인트는 테스트와 함께 작성
```

**💡 포인트**: "어떻게" 작성할지가 아니라 "무엇을" 담을지를 지시했다. Claude Code는 이 정보를 바탕으로 구조화된 CLAUDE.md를 생성할 것이다.

### 2.3 프로젝트 스캐폴딩

CLAUDE.md가 생성되면 프로젝트 전체 구조를 만들도록 지시한다.

```
이 프로젝트의 전체 디렉토리 구조를 만들어줘.

백엔드와 프론트엔드를 분리하고, Docker Compose로 통합 개발 환경을 구성해줘.
다음을 포함해야 해:

1. backend/ - FastAPI 앱 (app/api, app/models, app/schemas, app/services, app/core 구조)
2. frontend/ - Next.js 앱
3. docker/ - Dockerfile들
4. docker-compose.yml - 백엔드, 프론트엔드, PostgreSQL 서비스
5. .env.example - 환경 변수 템플릿
6. 각 서비스의 기본 설정 파일 (pyproject.toml, package.json 등)
```

Claude Code는 이 지시를 받으면 파일 시스템에 직접 디렉토리와 파일을 생성한다. 생성 과정에서 각 단계를 승인해야 할 수 있다 — 이것은 Claude Code의 안전 장치다.

### 2.4 서브에이전트 정의

프로젝트 구조가 잡히면 이후 Phase에서 사용할 서브에이전트를 미리 정의한다.

```
.claude/agents/ 디렉토리를 만들고 다음 서브에이전트를 정의해줘.

중요: 각 파일은 반드시 맨 앞에 YAML frontmatter(name, description, tools)를 포함해야 해.
Frontmatter가 없으면 서브에이전트로 인식되지 않아.

1. backend-developer.md
   - FastAPI 백엔드 전문
   - tools: Read, Write, Edit, Bash, Glob, Grep

2. frontend-developer.md
   - Next.js 프론트엔드 전문
   - tools: Read, Write, Edit, Bash, Glob, Grep

3. qa-engineer.md
   - 테스트 작성 및 품질 검증
   - tools: Read, Write, Edit, Bash, Glob, Grep

4. code-reviewer.md
   - 코드 리뷰 (읽기 전용)
   - tools: Read, Grep, Glob

5. devops-engineer.md
   - Docker, CI/CD 설정
   - tools: Read, Write, Edit, Bash, Glob, Grep

각 에이전트의 역할, 규칙, 출력 형식을 상세하게 작성해줘.
```

## 3. Phase 2: 백엔드 구현

### 📝 프롬프트 작성 가이드: 상세 vs 간략

Phase 2부터는 많은 섹션에서 **두 가지 프롬프트 옵션**을 제공한다. 상황에 맞게 선택하면 된다.

#### 상세 버전을 선택할 때:
- **예측 가능한 결과**가 필요할 때 (튜토리얼 따라하기)
- **특정 라이브러리나 패턴**을 명시하고 싶을 때
- **프로젝트의 핵심 구조**를 정의할 때 (데이터 모델, 아키텍처)
- **학습 목적**으로 정확한 결과를 확인하고 싶을 때

#### 간략 버전을 선택할 때:
- **에이전트의 판단을 신뢰**하고 더 나은 구현을 기대할 때
- **표준 패턴**(CRUD, JWT 인증, REST API)을 사용할 때
- **빠른 프로토타이핑**이 목적일 때
- **초보자**로서 위압적이지 않은 지시를 원할 때

**핵심**: 일관성보다는 각 작업의 성격에 맞게 선택하는 것이 중요하다.

---

### 3.1 데이터베이스 모델 및 마이그레이션

**옵션 1: 상세 버전** (모든 필드를 명시적으로 지정)

```
백엔드 데이터베이스 모델을 구현해줘.

모델: User, Project, ProjectMember, Task, Comment

요구사항:
- SQLAlchemy 2.0 스타일 (Mapped, mapped_column)
- User: email(unique), hashed_password, name, created_at
- Project: name, description, owner_id(FK→User), created_at
- ProjectMember: user_id(FK), project_id(FK), role(enum: owner/admin/member)
- Task: title, description, status(enum: todo/in_progress/done),
  priority(enum: low/medium/high/critical), project_id(FK),
  assignee_id(FK→User, nullable), created_at, updated_at
- Comment: content, task_id(FK), author_id(FK→User), created_at

Alembic 마이그레이션도 설정하고 초기 마이그레이션을 생성해줘.
```

**옵션 2: 간략 버전** (핵심만 전달하고 세부사항은 에이전트에게 위임)

```
백엔드 데이터베이스 모델을 구현해줘.

필요한 모델: User, Project, ProjectMember, Task, Comment

주요 기능:
- User: 사용자 정보 (이메일, 비밀번호, 이름)
- Project: 프로젝트 정보와 소유자 관계
- ProjectMember: 프로젝트 멤버십과 역할 (owner/admin/member)
- Task: 태스크 정보 (제목, 설명, 상태, 우선순위, 담당자)
  * status는 todo/in_progress/done으로 해줘
- Comment: 태스크에 대한 댓글

Alembic으로 마이그레이션 설정도 함께 해줘.
```

**💡 에이전틱 코딩 팁**: 옵션 2처럼 간략하게 지시해도 Claude Code는 실무에 적합한 필드들을 자동으로 추가한다. "무엇을" 만들지에 집중하고 "어떻게"는 에이전트에게 맡기는 것이 에이전틱 코딩의 핵심이다.

### 3.2 인증 시스템

**옵션 1: 상세 버전**

```
JWT 기반 인증 시스템을 구현해줘.

포함할 것:
- POST /api/auth/register - 회원가입
- POST /api/auth/login - 로그인 (JWT 토큰 발급)
- GET /api/auth/me - 현재 사용자 정보
- 비밀번호 해싱: passlib + bcrypt
- JWT: python-jose
- Pydantic 스키마 정의
- 의존성 주입으로 현재 사용자 가져오기 (get_current_user)

테스트도 함께 작성해줘 (pytest + httpx AsyncClient)
```

**옵션 2: 간략 버전**

```
JWT 기반 인증 시스템을 구현해줘.

필요한 기능:
- 회원가입 (POST /api/auth/register)
- 로그인 (POST /api/auth/login) - JWT 토큰 발급
- 현재 사용자 정보 조회 (GET /api/auth/me)
- 비밀번호 해싱 처리
- 인증이 필요한 엔드포인트에서 사용할 의존성 함수

테스트도 함께 작성해줘.
```

### 3.3 핵심 API 엔드포인트

**옵션 1: 상세 버전**

```
다음 API를 구현해줘. 각 API는 Pydantic 스키마, 서비스 레이어, 라우터, 테스트를 모두 포함해야 해.

프로젝트 API:
- POST /api/projects - 프로젝트 생성
- GET /api/projects - 내 프로젝트 목록
- GET /api/projects/{id} - 프로젝트 상세
- PUT /api/projects/{id} - 프로젝트 수정
- DELETE /api/projects/{id} - 프로젝트 삭제
- POST /api/projects/{id}/members - 멤버 추가

태스크 API:
- POST /api/projects/{project_id}/tasks - 태스크 생성
- GET /api/projects/{project_id}/tasks - 태스크 목록 (필터링, 정렬)
- GET /api/tasks/{id} - 태스크 상세
- PUT /api/tasks/{id} - 태스크 수정
- PATCH /api/tasks/{id}/status - 상태 변경
- DELETE /api/tasks/{id} - 태스크 삭제

댓글 API:
- POST /api/tasks/{task_id}/comments - 댓글 작성
- GET /api/tasks/{task_id}/comments - 댓글 목록

모든 엔드포인트는 인증이 필요하고, 프로젝트 멤버만 접근 가능해야 해.
```

**옵션 2: 간략 버전**

```
프로젝트와 태스크 관리를 위한 REST API를 구현해줘.

프로젝트 API:
- CRUD (생성, 목록, 상세, 수정, 삭제)
- 멤버 추가 기능

태스크 API:
- CRUD (생성, 목록, 상세, 수정, 삭제)
- 상태 변경 (PATCH)
- 목록 조회 시 필터링과 정렬 지원

댓글 API:
- 댓글 작성 및 목록 조회

모든 엔드포인트는 인증 필요, 프로젝트 멤버만 접근 가능하게 해줘.
각 API는 Pydantic 스키마, 서비스 레이어, 라우터, 테스트를 포함해야 해.
```

### 3.4 Plan Mode 활용하기

복잡한 구현을 지시하기 전에 **Plan Mode**를 활용하면 에이전트가 먼저 계획을 세우고, 개발자가 검토한 후에 실행으로 넘어갈 수 있다.

```
# Claude Code에서 Shift+Tab 두 번 → Plan Mode 전환

태스크 API의 필터링과 정렬 기능을 어떻게 구현할 계획인지 설명해줘.
쿼리 파라미터 설계, 서비스 레이어 로직, 페이지네이션 전략을 포함해서.
```

Plan Mode에서 계획을 확인하고 수정한 뒤, 다시 일반 모드로 전환하여 실행을 지시한다.

### 3.5 테스트 실행 및 수정

```
백엔드 테스트를 전부 실행하고, 실패하는 테스트가 있으면 수정해줘.
Docker Compose로 PostgreSQL을 띄운 상태에서 테스트해야 해.
```

Claude Code는 테스트를 실행하고, 실패한 테스트를 분석하여 코드를 수정하는 과정을 자율적으로 반복한다. 이것이 에이전틱 코딩의 핵심적인 강점이다 — 반복적인 디버깅 사이클을 에이전트가 자동으로 수행한다.

**✅ Phase 2 체크포인트**

## 5. Phase 4: 멀티 에이전트 협업 실습

**이 Phase가 이 튜토리얼의 핵심이다.** 앞서 정의한 서브에이전트와 에이전트 팀을 활용하여 여러 에이전트가 협업하는 워크플로우를 실습한다.

### 5.1 시나리오: 새로운 기능 추가를 멀티 에이전트로 수행하기

다음 기능을 추가한다고 가정하자: **"태스크에 마감일(due_date)과 라벨(labels) 기능 추가"**

이 기능은 백엔드 스키마 변경, API 수정, 프론트엔드 UI 업데이트, 테스트 작성이 모두 필요하므로 멀티 에이전트 협업의 좋은 예시다.

### 5.2 방법 1: 서브에이전트를 활용한 순차 협업

하나의 메인 세션에서 서브에이전트에게 작업을 위임하는 방식이다.

```
태스크에 두 가지 새 기능을 추가하려고 해:
1. 마감일 (due_date: datetime, nullable)
2. 라벨 (labels: 다대다 관계, Label 모델 새로 생성)

다음 순서로 서브에이전트를 사용해서 진행해줘:

1. backend-developer 서브에이전트로:
   - Task 모델에 due_date 필드 추가
   - Label 모델과 task_labels 연결 테이블 생성
   - Alembic 마이그레이션 생성 및 실행
   - 관련 Pydantic 스키마 업데이트
   - API 엔드포인트 수정/추가 (라벨 CRUD, 태스크에 라벨 연결)

2. qa-engineer 서브에이전트로:
   - 새 기능에 대한 테스트 코드 작성
   - 기존 테스트가 깨지지 않는지 확인

3. frontend-developer 서브에이전트로:
   - TaskCard에 마감일 표시 (기한 임박 시 색상 변경)
   - 라벨 뱃지 표시
   - CreateTaskForm에 마감일 선택기와 라벨 선택 추가
   - TaskDetailModal에 마감일/라벨 편집 기능 추가

4. code-reviewer 서브에이전트로:
   - 전체 변경 사항에 대한 코드 리뷰 실행
   - 보안, 성능, 코드 품질 관점으로 검토

각 단계의 결과를 보여주고, 다음 단계로 넘어가기 전에 확인해줘.
```

**이 방식의 장점**: 순서가 보장되고, 각 단계의 결과를 확인할 수 있다. 의존 관계가 있는 작업(예: 백엔드 완성 후 프론트엔드 작업)에 적합하다.

### 5.3 방법 2: 에이전트 팀을 활용한 병렬 협업

에이전트 팀을 사용하면 여러 작업을 동시에 진행할 수 있다.

```
에이전트 팀을 만들어서 다음 작업을 병렬로 진행해줘:

팀 구성:
- 백엔드 개발자: Task 모델에 due_date 추가, Label 모델 생성,
  API 엔드포인트 수정
- 프론트엔드 개발자: 현재 API 스펙을 기반으로 UI 컴포넌트 업데이트
  (마감일 표시, 라벨 뱃지)
- QA 엔지니어: 기존 테스트 확인 및 새 기능 테스트 작성

백엔드 개발자가 스키마 변경을 완료하면 프론트엔드 개발자에게
알려서 타입을 업데이트하도록 해줘.
QA 엔지니어는 각 개발자의 작업이 끝날 때마다 테스트를 실행해줘.
```

**이 방식의 장점**: 독립적으로 진행 가능한 작업을 병렬로 수행하여 시간을 절약한다. 팀원 간 메시지 교환으로 의존 관계도 처리할 수 있다.

**주의 사항**: 에이전트 팀은 토큰 사용량이 팀원 수에 비례하여 증가한다. 단순한 작업에는 오버헤드가 크므로, 병렬 탐색이 실질적인 가치를 제공하는 경우에만 사용하라.

### 5.4 방법 3: 하이브리드 워크플로우 (권장)

실무에서는 서브에이전트와 에이전트 팀을 상황에 따라 조합하는 것이 가장 효과적이다.

**1단계: 설계 (메인 세션, Plan Mode)**

```
[Plan Mode] 태스크에 마감일과 라벨 기능을 추가하려고 해.
백엔드, 프론트엔드, 테스트 각각에서 필요한 변경 사항을 정리하고,
작업 간 의존 관계를 분석해줘.
어떤 작업을 병렬로 진행할 수 있는지도 알려줘.
```

**2단계: 병렬 구현 (에이전트 팀)**

```
에이전트 팀을 만들어서 방금 계획한 대로 구현을 진행해줘.

- 백엔드 팀원: 모델 + 마이그레이션 + API 수정
- 프론트엔드 팀원: 컴포넌트 업데이트 + 타입 정의
- QA 팀원: 테스트 작성 + 실행

각 팀원은 작업 완료 시 팀 리드에게 보고해줘.
```

**3단계: 코드 리뷰 (서브에이전트)**

```
code-reviewer 서브에이전트를 사용해서 이번에 변경된 모든 파일을
리뷰해줘. 보안, 성능, 코드 품질 관점에서 검토하고,
심각도별로 이슈를 정리해줘.
```

**4단계: 수정 및 최종 확인 (메인 세션)**

```
리뷰에서 나온 이슈들을 수정해줘.
수정 후 전체 테스트를 다시 실행해서 모두 통과하는지 확인해줘.
```

**5단계: 커밋 및 정리**

```
변경 사항을 의미 있는 커밋 단위로 나눠서 커밋해줘.
커밋 메시지는 Conventional Commits 형식으로.
```

**✅ Phase 4 체크포인트**

- [ ] 태스크에 due_date 필드 추가 (백엔드)
- [ ] Label 모델 생성 및 다대다 관계 설정 (백엔드)
- [ ] 마이그레이션 생성 및 적용
- [ ] API 엔드포인트 업데이트 (라벨 CRUD)

- [ ] 테스트 코드 작성 및 통과
- [ ] 코드 리뷰 완료

## 6. Phase 5: DevOps 및 QA

### 6.1 Docker 환경 최적화

```
devops-engineer 서브에이전트를 사용해서 다음을 수행해줘:

1. backend/Dockerfile 최적화
   - 멀티스테이지 빌드
   - .dockerignore 설정
   - 비루트 사용자 설정

2. frontend/Dockerfile 최적화
   - Next.js standalone 출력 모드
   - 멀티스테이지 빌드

3. docker-compose.yml 업데이트
   - 헬스체크 추가
   - 볼륨 설정 검증
   - 환경 변수 관리 개선
```

### 6.2 CI/CD 파이프라인

```
GitHub Actions 워크플로우 파일을 만들어줘.

.github/workflows/ci.yml:
- PR 생성 시 트리거
- 백엔드 린트 (ruff) + 테스트 (pytest)
- 프론트엔드 린트 (eslint) + 타입 체크 + 빌드
- PostgreSQL 서비스 컨테이너 사용
```

### 6.3 최종 통합 테스트

```
qa-engineer 서브에이전트를 사용해서 전체 통합 테스트를 수행해줘.

1. Docker Compose로 전체 서비스를 올려서 E2E 테스트
2. 주요 사용자 시나리오 테스트:
   - 회원가입 → 로그인 → 프로젝트 생성 → 태스크 생성
     → 상태 변경 → 댓글 작성
3. 에러 케이스 테스트:
   - 인증 없이 API 접근
   - 권한 없는 프로젝트/태스크 접근
   - 잘못된 입력값
4. 테스트 결과 보고서 생성
```
