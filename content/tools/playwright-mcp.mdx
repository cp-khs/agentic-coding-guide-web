## 1. Playwright MCP란?

Playwright MCP는 Claude Code가 실제 브라우저를 직접 제어할 수 있게 해주는 MCP 서버입니다. Microsoft가 공식 운영하며 `https://mcp.playwright.dev/mcp` HTTP 엔드포인트를 통해 연결합니다.

### 핵심 가치

```
[기존 방식]
브라우저 에러 발생
    ↓
개발자가 F12 → DevTools 열기
    ↓
에러 메시지 복사
    ↓
Claude Code에 붙여넣기
    ↓
수정

[Playwright MCP 방식]
브라우저 에러 발생
    ↓
"http://localhost:3000 접속해서 콘솔 에러 확인 후 수정해줘"
    ↓
Claude Code가 직접 접속 → 에러 캡처 → 코드 수정 → 재확인
```

### stdio vs HTTP 방식 이해

Playwright MCP는 **stdio(npx) 방식**을 사용합니다. `@playwright/mcp` npm 패키지가 로컬에서 실행되며, Node.js >= 18을 요구하여 Claude Code 내장 Node.js(v18.18.1)와 호환됩니다.

```json
{
  "type": "stdio",
  "command": "/usr/local/bin/npx",
  "args": ["@playwright/mcp@latest"]
}
```

## 2. 설치 및 등록

### 사전 준비: 전역 설치

`npx` 즉시 실행 방식은 Claude Code 샌드박스 환경에서 `sh` 찾기 실패 문제가 발생합니다.
반드시 **전역 설치** 후 바이너리 절대 경로를 사용해야 합니다.

```bash
npm install -g @playwright/mcp@latest
```

### 전역 등록 (모든 프로젝트에서 사용)

```bash
claude mcp add --scope user playwright -- /usr/local/bin/playwright-mcp
```

### 프로젝트 한정 등록

```bash
claude mcp add --scope project playwright -- /usr/local/bin/playwright-mcp
```

### 등록 확인

```bash
claude mcp list
```

`~/.claude.json`에 다음과 같이 저장됩니다:

```json
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "/usr/local/bin/playwright-mcp",
      "args": [],
      "env": {}
    }
  }
}
```

## 3. 주요 도구 목록

| 도구 | 설명 | 주요 용도 |
|------|------|-----------|
| `browser_navigate` | URL로 이동 | 페이지 접속 |
| `browser_screenshot` | 현재 화면 캡처 | 렌더링 확인 |
| `browser_click` | 요소 클릭 | 버튼, 링크 동작 테스트 |
| `browser_type` | 텍스트 입력 | 폼 작성 |
| `browser_select_option` | 드롭다운 선택 | 옵션 선택 |
| `browser_console_messages` | 콘솔 로그 수집 | JS 에러 확인 |
| `browser_network_requests` | 네트워크 요청 목록 | API 호출 확인 |
| `browser_evaluate` | JavaScript 실행 | DOM/상태 조회 |
| `browser_wait_for` | 조건 대기 | 비동기 로딩 대기 |
| `browser_hover` | 마우스 호버 | 툴팁, 드롭다운 메뉴 |
| `browser_navigate_back` | 뒤로가기 | 페이지 흐름 테스트 |
| `browser_close` | 브라우저 종료 | 세션 정리 |

## 4. 실전 사용 시나리오

### 시나리오 1: 콘솔 에러 자동 수정

개발 중 화이트스크린이 발생했을 때:

```
"http://localhost:3000/settlements 에 접속해서 콘솔 에러를 확인하고,
에러가 있으면 원인 파일을 찾아서 수정해줘."
```

Claude Code가 수행하는 과정:
1. `browser_navigate` → `http://localhost:3000/settlements`
2. `browser_console_messages` → 에러 메시지 수집
3. 에러 스택 트레이스로 파일 위치 파악
4. 코드 수정
5. `browser_screenshot` → 수정 결과 확인

---

### 시나리오 2: API 연동 오류 디버깅

React ↔ FastAPI 통신 문제:

```
"http://localhost:3000 에서 통행료 계산 버튼을 클릭했을 때
발생하는 네트워크 요청을 확인해줘. 특히 실패한 요청이 있으면
백엔드 코드를 찾아서 수정해줘."
```

Claude Code가 수행하는 과정:
1. `browser_navigate` → `http://localhost:3000`
2. `browser_click` → 계산 버튼 클릭
3. `browser_network_requests` → API 요청/응답 확인
4. 실패한 요청의 상태 코드(404, 500 등) 파악
5. 백엔드 라우터/서비스 코드 수정

---

### 시나리오 3: 폼 입력 및 결과 검증

```
"http://localhost:3000/verify 에 접속해서
- 시작 IC: 서울
- 도착 IC: 부산
- 차종: 승용차
를 입력하고 검증 버튼을 클릭한 뒤, 결과 테이블의 내용을 확인해줘."
```

---

### 시나리오 4: E2E 테스트 시나리오 자동 작성

```
"http://localhost:3000 의 로그인부터 정산 내역 조회까지
사용자 흐름을 브라우저로 직접 실행해보고,
동일한 흐름을 pytest + playwright 테스트 코드로 작성해줘."
```

---

### 시나리오 5: 반응형 레이아웃 확인

```
"http://localhost:3000/dashboard 를 다음 화면 크기로 각각 스크린샷 찍어줘:
- 모바일: 375 x 812
- 태블릿: 768 x 1024
- 데스크탑: 1440 x 900
레이아웃 깨짐이 있으면 CSS를 수정해줘."
```

---

### 시나리오 6: JavaScript 상태 직접 조회

```
"http://localhost:3000/calculator 에서 React 컴포넌트의
현재 state를 확인해줘. 특히 tollAmount 값이 올바른지 봐줘."
```

`browser_evaluate`로 React DevTools 없이 상태 조회:

```javascript
// Claude Code가 내부적으로 실행하는 코드 예시
window.__REACT_DEVTOOLS_GLOBAL_HOOK__?.renderers
```

## 5. 샘플 프롬프트 모음

### 기본 탐색 및 확인

```
"http://localhost:3000 에 접속해서 스크린샷을 찍어줘."
```

```
"http://localhost:3000 의 콘솔에 에러가 있는지 확인해줘."
```

```
"http://localhost:3000 에서 모든 네트워크 요청 목록을 보여줘.
실패한 요청(4xx, 5xx)이 있으면 알려줘."
```

---

### 자동 디버깅

```
"http://localhost:3000/[문제 페이지] 에 접속해서
1. 콘솔 에러 확인
2. 네트워크 실패 요청 확인
3. 발견된 문제를 코드에서 찾아 수정
4. 수정 후 스크린샷으로 결과 확인
위 순서대로 진행해줘."
```

---

### 폼 자동화

```
"http://localhost:3000/[폼 페이지] 에서 다음 데이터로 폼을 작성하고 제출해줘:
- 필드명1: 값1
- 필드명2: 값2
제출 후 성공/에러 메시지를 확인하고 스크린샷을 찍어줘."
```

---

### 회귀 테스트

```
"다음 페이지들을 순서대로 방문하면서 각각 스크린샷을 찍어줘.
레이아웃 깨짐이나 콘솔 에러가 있으면 보고해줘:
- http://localhost:3000/
- http://localhost:3000/routes
- http://localhost:3000/settlements
- http://localhost:3000/simulator"
```

---

### CI/CD 연동 (헤드리스 모드)

Playwright MCP는 헤드리스 브라우저로도 동작합니다. GitHub Actions 등 CI 환경에서 활용 가능합니다.

```yaml
# .github/workflows/e2e.yml 예시
- name: Run E2E via Claude Code + Playwright MCP
  run: |
    claude --headless "http://localhost:3000 에 접속해서
    핵심 페이지 5개를 방문하고 에러가 있으면 보고서를 e2e-report.md에 저장해줘."
```

## 6. 기존 방식과 비교

| 항목 | 기존 방식 | Playwright MCP |
|------|-----------|----------------|
| 에러 확인 | F12 → 복사 → 붙여넣기 | 자동 수집 |
| API 디버깅 | Network 탭 수동 확인 | 자동 캡처 |
| 반복 테스트 | 매번 수동 | 자동화 |
| CI 연동 | 별도 Playwright 설정 | MCP 하나로 통합 |
| 학습 곡선 | Playwright API 학습 필요 | 자연어 프롬프트 |

## 7. 주의사항 및 팁

### 로컬 서버 먼저 실행

Playwright MCP가 접속할 수 있도록 로컬 서버가 먼저 실행 중이어야 합니다:

```bash
# 예시: Docker Compose 기반 프로젝트
docker-compose up -d

# 또는 개발 서버
npm run dev       # 프론트엔드
poetry run uvicorn app.main:app --reload  # 백엔드
```

---

### 인증이 필요한 페이지

로그인이 필요한 페이지는 먼저 로그인 과정을 프롬프트에 포함시킵니다:

```
"http://localhost:3000/login 에서
- 이메일: admin@example.com
- 비밀번호: admin123
으로 로그인한 뒤, /dashboard 페이지의 스크린샷을 찍어줘."
```

---

### 비동기 로딩 처리

데이터 로딩 완료를 기다리는 프롬프트 작성법:

```
"http://localhost:3000/settlements 에 접속하고
데이터 테이블이 로딩될 때까지 기다렸다가 스크린샷을 찍어줘.
테이블에 데이터가 없으면 이유를 네트워크 요청에서 확인해줘."
```

---

### 스크린샷 저장

스크린샷을 파일로 저장하려면:

```
"http://localhost:3000 의 스크린샷을 찍어서
./screenshots/homepage-$(date +%Y%m%d).png 로 저장해줘."
```

---

### Playwright MCP vs LSP 진단의 차이

| 구분 | LSP 진단 | Playwright MCP |
|------|----------|----------------|
| 실행 시점 | 코드 작성 중 (정적) | 브라우저 실행 중 (동적) |
| 탐지 대상 | 타입 오류, 문법 오류 | 런타임 에러, API 오류 |
| 환경 | 에디터 | 실제 브라우저 |
| 예시 | `str은 int에 할당 불가` | `404 Not Found: /api/routes` |

두 도구는 **상호 보완** 관계입니다. LSP가 코드 수준 오류를 잡고, Playwright MCP가 실행 환경 오류를 잡습니다.
