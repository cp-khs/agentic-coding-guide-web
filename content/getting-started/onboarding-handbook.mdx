## 1. 에이전틱 코딩이란?

### 1.1 AI 코딩 방식의 진화 — 왜 지금 에이전틱 코딩인가

AI를 활용한 코딩 방식은 네 단계를 거쳐 진화해왔다. 에이전틱 코딩이 어디에서 왔고, 기존 방식과 어떻게 다른지를 이해하면 도입의 맥락이 명확해진다.

**1단계: 코드 자동완성 (2021~)**

GitHub Copilot이 대표적이다. 개발자가 코드를 작성하면 다음 줄을 예측해서 제안하는 방식이다. 타이핑 속도를 높여주지만, 모든 결정은 여전히 개발자가 내린다.

**2단계: 대화형 코딩 어시스턴트 (2023~)**

ChatGPT, Claude.ai 같은 챗봇에 코드를 붙여넣고 질문하는 방식이다. "이 코드에서 버그를 찾아줘", "이 함수를 리팩토링해줘"처럼 사용한다. 문맥 이해력은 올라갔지만, 복사-붙여넣기 과정이 필수이고 실행 환경과 분리되어 있다는 한계가 있다.

**3단계: IDE 통합 AI (2024~)**

Cursor, Windsurf 같은 AI 네이티브 IDE가 등장한다. AI가 프로젝트 파일을 직접 읽고, 에디터 안에서 코드를 수정한다. 복사-붙여넣기 없이 작업할 수 있게 됐지만, 여전히 개발자가 파일 단위로 작업을 지시하는 구조다.

**4단계: 에이전틱 코딩 (2025~)**

Claude Code, OpenAI Codex 등이 여기에 해당한다. AI가 단순히 코드를 제안하는 게 아니라, 스스로 계획을 세우고, 파일을 생성하고, 명령어를 실행하고, 테스트를 돌리고, 실패하면 수정하는 전체 사이클을 자율적으로 수행한다. 더 나아가 여러 에이전트가 역할을 나눠 병렬로 협업하는 멀티 에이전트 방식까지 발전했다.

```
┌────────────────────────────────────────────────────────────────┐
│                   AI 코딩 방식의 진화                            │
│                                                                │
│  1단계        2단계         3단계          4단계                 │
│  자동완성 →   대화형 →      IDE 통합 →     에이전틱              │
│  (2021~)     (2023~)      (2024~)       (2025~)               │
│                                                                │
│  다음 줄     코드 Q&A      파일 직접      자율적 계획·실행·검증   │
│  예측        복사-붙여넣기   읽기/수정      멀티 에이전트 협업      │
│                                                                │
│  개발자 역할의 변화:                                             │
│  타이핑 → 질문하기 → 지시하기 → 오케스트레이션                    │
└────────────────────────────────────────────────────────────────┘
```

이 핸드북은 4단계인 에이전틱 코딩, 그중에서도 Claude Code를 활용한 멀티 에이전트 협업에 초점을 맞춘다.

### 1.2 기존 코딩 방식과의 구체적 차이

기존에 우리가 사용해온 AI 코딩 도구(GitHub Copilot, ChatGPT 등)는 "자동완성" 혹은 "질의응답" 패러다임이었다. 개발자가 주도권을 쥐고, AI는 보조 역할을 했다.

에이전틱 코딩은 근본적으로 다르다. **AI 에이전트가 자율적으로 코드베이스를 탐색하고, 파일을 생성·수정하며, 명령어를 실행하고, 테스트를 수행한다.** 개발자는 "무엇을 만들 것인가"를 지시하고, 에이전트는 "어떻게 만들 것인가"를 스스로 결정하여 실행한다.

```
┌─────────────────────────────────────────────────────┐
│              기존 AI 코딩 도구                        │
│  개발자 → [코드 작성] → AI가 자동완성 제안 → 수락/거절  │
│  개발자가 모든 결정을 내림                              │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│              에이전틱 코딩                             │
│  개발자 → [목표 지시] → AI가 계획 수립 → 코드 작성     │
│         → 테스트 실행 → 오류 수정 → 결과 보고          │
│  개발자는 방향과 품질을 관리                            │
└─────────────────────────────────────────────────────┘
```

### 1.3 경험 많은 개발자가 알아야 할 마인드셋 전환

10년 이상 경력의 개발자가 에이전틱 코딩을 시작할 때 가장 흔히 겪는 어려움은 **"통제권을 내려놓는 것"**이다. 몇 가지 마인드셋 전환이 필요하다.

**"어떻게(How)" 대신 "무엇을(What)" 지시하기**: "FastAPI에서 Pydantic 모델로 User 스키마를 만들고 SQLAlchemy ORM으로 CRUD를 구현해줘"보다 "사용자 관리 API를 만들어줘. 이메일 인증, 프로필 수정, 비밀번호 변경 기능이 필요해"가 더 좋은 지시다. 에이전트에게 구현 결정권을 줄수록 더 나은 결과를 얻는다.

**코드 리뷰어로 역할 전환**: 직접 코드를 작성하는 대신, 에이전트가 생성한 코드를 리뷰하고, 방향을 수정하고, 품질 기준을 설정하는 역할에 집중한다. 이것은 시니어 개발자가 주니어를 멘토링하는 것과 유사하다.

**반복적 대화로 정밀도 높이기**: 한 번의 완벽한 프롬프트보다 여러 번의 대화를 통해 점진적으로 원하는 결과에 도달하는 것이 효과적이다. 에이전트에게 피드백을 주고, 수정을 요청하고, 대안을 탐색하는 과정이 자연스럽다.

## 2. Claude Code 소개 및 설치

### 2.1 Claude Code란?

Claude Code는 Anthropic에서 만든 터미널 기반 AI 코딩 에이전트다. 단순한 코드 생성기가 아니라, 프로젝트 전체를 이해하고 자율적으로 작업하는 에이전트로 동작한다.

주요 특징은 다음과 같다.

- **터미널 네이티브**: VS Code, JetBrains 등과 통합 가능하지만, 기본적으로 터미널에서 동작한다
- **코드베이스 인식**: 프로젝트 전체 구조를 파악하고, 파일 간 관계를 이해한다
- **자율 실행**: 파일 읽기/쓰기, 쉘 명령 실행, Git 작업 등을 직접 수행한다
- **CLAUDE.md**: 프로젝트별 컨텍스트와 규칙을 설정하는 설정 파일
- **서브에이전트**: 특정 작업을 위한 전문화된 하위 에이전트를 생성하여 위임할 수 있다
- **에이전트 팀**: 여러 Claude Code 인스턴스가 병렬로 협업하는 실험적 기능

### 2.2 설치

Claude Code 설치는 간단하다. 공식 권장 방법은 네이티브 바이너리 설치다.

```bash
# macOS / Linux (권장)
curl -fsSL https://claude.ai/install.sh | bash

# 설치 후 셸 설정 리로드
source ~/.bashrc   # bash 사용자
source ~/.zshrc    # zsh 사용자

# 설치 확인
claude --version
claude doctor      # 설치 상태 진단
```

> **참고 링크**
> - 공식 설치 가이드: https://code.claude.com/docs/en/setup
> - 트러블슈팅 가이드: https://code.claude.com/docs/en/troubleshooting

### 2.3 인증 및 요금제

Claude Code를 사용하려면 다음 중 하나가 필요하다.

- **Claude Pro 또는 Max 플랜** (권장): claude.ai 계정으로 통합 로그인
- **Anthropic API 콘솔**: API 키 기반 인증 (console.anthropic.com)
- **Claude for Teams / Enterprise**: 팀 단위 중앙 관리

```bash
# 프로젝트 디렉토리에서 Claude Code 시작
cd your-project
claude

# 첫 실행 시 인증 프로세스가 자동으로 시작된다
```

### 2.4 모델 선택

Claude Code에서 사용할 모델을 선택할 수 있다. 에이전틱 코딩의 품질에 직접적인 영향을 미치므로 적절한 모델을 선택하는 것이 중요하다.

| 모델 | 특징 | 권장 용도 |
|------|------|-----------|
| Claude Opus 4.6 | 최고 성능, 1M 토큰 컨텍스트, 에이전트 팀 지원 | 복잡한 아키텍처 설계, 멀티 에이전트 작업 |
| Claude Sonnet 4.5 | 성능과 속도의 균형 | 일반 개발, 일상적 코딩 작업 |
| Claude Haiku 4.5 | 빠른 응답 | 간단한 수정, 빠른 질의 |

```bash
# 모델 설정 (환경 변수)
export ANTHROPIC_MODEL="claude-sonnet-4-5-20250929"
```

## 3. 핵심 개념

### 3.1 CLAUDE.md — 프로젝트의 "두뇌"

CLAUDE.md는 Claude Code가 프로젝트를 이해하기 위해 참조하는 마크다운 파일이다. 프로젝트 루트에 위치하며, 모든 세션에서 자동으로 로드된다. 경험 많은 개발자에게 비유하자면, 새로운 팀원에게 전달하는 "프로젝트 온보딩 문서"와 같다.

```markdown
# CLAUDE.md 예시

## 프로젝트 개요
TaskFlow - 팀 태스크 관리 웹 애플리케이션

## 기술 스택
- 백엔드: Python 3.12, FastAPI, SQLAlchemy, Alembic
- 프론트엔드: Next.js 15, TypeScript, Tailwind CSS
- 데이터베이스: PostgreSQL 16
- 인프라: Docker, Docker Compose

## 프로젝트 구조
backend/          # FastAPI 애플리케이션
  app/
    api/          # API 라우터
    models/       # SQLAlchemy 모델
    schemas/      # Pydantic 스키마
    services/     # 비즈니스 로직
    core/         # 설정, 인증, DB 연결
frontend/         # Next.js 애플리케이션
  src/
    app/          # App Router 페이지
    components/   # React 컴포넌트
    lib/          # 유틸리티, API 클라이언트
docker/           # Docker 관련 파일

## 코딩 컨벤션
- 백엔드: PEP 8 준수, type hint 필수, docstring 작성
- 프론트엔드: ESLint + Prettier, 함수형 컴포넌트만 사용
- 커밋 메시지: Conventional Commits 형식 (feat:, fix:, docs: 등)
- 브랜치 전략: main → develop → feature/xxx

## 중요 규칙
- 코드 변경 전 반드시 git branch를 생성할 것
- API 엔드포인트는 반드시 테스트 코드와 함께 작성할 것
- 환경 변수는 .env 파일로 관리하고, 절대 하드코딩하지 않을 것
- 모든 API 응답은 일관된 형식(ApiResponse)을 따를 것
```

**CLAUDE.md 작성 팁**: 너무 길면 오히려 역효과다. 핵심 구조, 컨벤션, 금지 사항에 집중하고, 상세한 문서는 별도 파일로 분리하여 링크하는 것이 좋다.

### 3.2 서브에이전트 — 전문화된 작업 위임

서브에이전트는 특정 작업에 특화된 하위 에이전트다. 메인 세션의 컨텍스트를 오염시키지 않고 독립적인 작업을 수행한 뒤 결과만 반환한다.

```
┌─────────────────────────────────────┐
│         메인 Claude Code 세션        │
│                                     │
│  "사용자 인증 API를 구현해줘"         │
│         │                           │
│         ├──→ [서브에이전트: 백엔드]    │
│         │     JWT 인증 구현           │
│         │     ← 결과 반환             │
│         │                           │
│         ├──→ [서브에이전트: 테스트]    │
│         │     인증 API 테스트 작성     │
│         │     ← 결과 반환             │
│         │                           │
│         └──→ [서브에이전트: 리뷰]     │
│               보안 취약점 점검         │
│               ← 결과 반환             │
└─────────────────────────────────────┘
```

서브에이전트는 `.claude/agents/` 디렉토리에 마크다운 파일로 정의한다.

```markdown
{/* .claude/agents/backend-developer.md */}
---
name: backend-developer
description: FastAPI 백엔드 개발 전문 에이전트
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Glob
  - Grep
---

# 백엔드 개발자 에이전트

## 역할
FastAPI 기반 백엔드 API를 설계하고 구현합니다.

## 규칙
- 모든 엔드포인트에 Pydantic 스키마를 사용합니다
- SQLAlchemy ORM으로 데이터베이스를 조작합니다
- 비동기(async/await) 패턴을 사용합니다
- 에러 처리는 HTTPException을 활용합니다
- 각 엔드포인트에 대한 docstring을 작성합니다
```

```markdown
{/* .claude/agents/code-reviewer.md */}
---
name: code-reviewer
description: 코드 리뷰 전문 에이전트
tools:
  - Read
  - Grep
  - Glob
---

# 코드 리뷰어 에이전트

## 역할
코드 품질, 보안, 성능을 검토합니다.

## 검토 항목
- 보안 취약점 (SQL 인젝션, XSS, 인증 우회 등)
- 성능 문제 (N+1 쿼리, 불필요한 연산 등)
- 코드 컨벤션 준수 여부
- 에러 처리의 완전성
- 테스트 커버리지

## 출력 형식
각 이슈를 심각도(Critical/Warning/Info)와 함께 보고합니다.
```

**서브에이전트 vs 직접 지시의 선택 기준**: 작업이 독립적이고 결과만 필요한 경우(예: 코드 리뷰, 테스트 작성) 서브에이전트가 적합하다. 반복적인 대화가 필요한 경우(예: 아키텍처 논의)는 메인 세션에서 직접 작업하는 것이 낫다.

### 3.3 에이전트 팀 — 병렬 협업의 핵심

에이전트 팀은 여러 Claude Code 인스턴스가 **병렬로** 작업하면서 서로 **소통**하는 기능이다. 서브에이전트와 달리 각 팀원이 독립적인 컨텍스트 윈도우를 가지며, 팀원 간 메시지를 주고받을 수 있다.

```
┌──────────────────────────────────────────────────┐
│                  에이전트 팀 구조                   │
│                                                  │
│  ┌──────────┐                                    │
│  │  팀 리드   │ ← 개발자가 지시                     │
│  │ (메인 세션) │                                   │
│  └────┬─────┘                                    │
│       │ 작업 배정 & 조율                            │
│       ├──────────────┬──────────────┐             │
│  ┌────▼─────┐  ┌─────▼────┐  ┌─────▼────┐       │
│  │ 팀원 A    │  │ 팀원 B   │  │ 팀원 C    │       │
│  │ 백엔드    │  │ 프론트엔드│  │ 테스트    │        │
│  │ 개발      │  │ 개발     │  │ 작성      │        │
│  └────┬─────┘  └─────┬────┘  └─────┬────┘       │
│       │              │              │             │
│       └──── 상호 메시지 교환 ─────────┘             │
│              (공유 태스크 리스트)                     │
└──────────────────────────────────────────────────┘
```

에이전트 팀은 현재 실험적(experimental) 기능이며, 활성화가 필요하다.

```bash
# 에이전트 팀 활성화 (환경 변수)
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# 또는 settings.json에 추가
# ~/.claude/settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

**서브에이전트 vs 에이전트 팀 비교**

| 항목 | 서브에이전트 | 에이전트 팀 |
|------|------------|------------|
| 실행 방식 | 단일 세션 내 순차/위임 | 복수 세션 병렬 실행 |
| 컨텍스트 | 메인 세션과 공유 (제한적) | 각자 독립 컨텍스트 윈도우 |
| 통신 | 결과만 메인으로 반환 | 팀원 간 직접 메시지 교환 |
| 토큰 사용량 | 상대적으로 적음 | 팀원 수에 비례하여 증가 |
| 적합한 작업 | 독립적·단발성 작업 | 병렬 탐색, 크로스 레이어 작업 |
| 안정성 | 안정적 | 실험적 (알려진 제한 사항 있음) |

## 4. 실전 팁과 안티패턴

### 4.1 효과적인 프롬프트 작성법

**좋은 프롬프트의 공통점**: 목표가 명확하고, 제약 조건이 구체적이며, 검증 가능한 결과물을 요구한다.

```
❌ 나쁜 예:
"API를 만들어줘"

✅ 좋은 예:
"태스크 상태를 변경하는 PATCH /api/tasks/{id}/status 엔드포인트를
만들어줘. 요청 본문은 {status: 'todo' | 'in_progress' | 'done'}이고,
프로젝트 멤버만 접근 가능해야 해. 상태 변경 시 updated_at을 자동
갱신하고, 유효하지 않은 상태값에 대해 422 에러를 반환해야 해.
pytest 테스트도 함께 작성해줘."
```

**좋은 프롬프트 체크리스트**:
- 무엇을 만들지 명확한가?
- 입력/출력이 정의되어 있는가?
- 제약 조건과 예외 케이스를 언급했는가?
- 검증 방법(테스트)을 요청했는가?

### 4.2 컨텍스트 관리

Claude Code의 컨텍스트 윈도우는 유한하다. 긴 세션에서는 컨텍스트가 소진될 수 있으므로 관리가 필요하다.

**`/clear` 명령어**: 대화 히스토리를 초기화하되 CLAUDE.md 컨텍스트는 유지한다. 새로운 작업을 시작할 때 유용하다.

**작업 단위로 세션 분리**: 하나의 세션에서 모든 것을 하려 하지 말고, "백엔드 API 구현" 세션, "프론트엔드 UI 구현" 세션으로 나누는 것이 효과적이다.

**서브에이전트 활용**: 탐색적 작업(코드 분석, 리뷰 등)은 서브에이전트에 위임하면 메인 세션의 컨텍스트를 절약할 수 있다.

### 4.3 흔한 안티패턴

| 안티패턴 | 문제 | 해결 |
|---------|------|------|
| 과도한 마이크로 관리 | 에이전트에게 한 줄씩 지시하면 비효율적 | 목표와 제약 조건을 주고 자율성을 부여 |
| CLAUDE.md 미작성 | 에이전트가 프로젝트 컨텍스트를 모름 | 프로젝트 시작 시 반드시 CLAUDE.md 작성 |
| 결과 미검증 | 에이전트의 출력을 무비판적으로 수용 | 각 단계마다 테스트 실행 및 코드 리뷰 |
| 단일 거대 프롬프트 | 너무 많은 것을 한 번에 요청 | 작업을 단계별로 나누어 지시 |
| 에이전트 팀 남용 | 단순 작업에 팀 구성 | 병렬 가치가 있을 때만 에이전트 팀 사용 |
| 수정 지시 누락 | 잘못된 방향으로 계속 진행 | 초기에 피드백하여 방향 수정 |

### 4.4 에이전트와의 효과적인 대화 패턴

**점진적 구체화**: 큰 그림에서 시작하여 세부 사항으로 좁혀간다.

```
1차: "사용자 인증 시스템을 만들어줘"
2차: "비밀번호 해싱에 bcrypt를 사용하고, 토큰 만료 시간은 30분으로 해줘"
3차: "리프레시 토큰도 추가하고, 로그아웃 시 토큰을 블랙리스트에 추가해줘"
```

**명시적 피드백**: 마음에 들지 않는 부분을 구체적으로 지적한다.

```
❌ "다시 해줘"
✅ "서비스 레이어의 에러 처리가 너무 단순해. 각 비즈니스 로직별로
    커스텀 예외를 정의하고, 적절한 HTTP 상태 코드를 매핑해줘."
```

**대안 탐색**: 에이전트에게 여러 접근 방식을 비교하도록 요청한다.

```
"태스크 필터링을 구현하는 방법으로 쿼리 파라미터 방식과
요청 본문 방식의 장단점을 비교해줘. 우리 프로젝트에 어떤 게 더 적합한지
추천해줘."
```

## 마무리: 에이전틱 코딩의 핵심 원칙

이 핸드북을 통해 학습한 내용을 정리하면, 에이전틱 코딩에서 경험 많은 개발자의 역할은 다음과 같이 전환된다.

**코드 작성자 → 아키텍트 + 리뷰어 + 오케스트레이터**

여러분의 10년 이상의 경험은 에이전틱 코딩에서 더 큰 가치를 발휘한다. 좋은 아키텍처를 설계하는 능력, 코드 품질을 판단하는 눈, 적절한 작업 분해 능력 — 이 모든 것이 에이전트를 효과적으로 활용하는 데 필수적이다.

에이전틱 코딩은 개발자를 대체하는 것이 아니라, **개발자의 영향력을 확장하는 도구**다. 한 명의 시니어 개발자가 여러 에이전트를 오케스트레이션하여 팀 규모의 결과물을 만들어낼 수 있다는 것이 핵심이다.
