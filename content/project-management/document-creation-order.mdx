# AI 코딩 프로젝트 문서 작성 순서 가이드

> **문서 목적**: AI 코딩 에이전트를 활용한 프로젝트에서 올바른 문서 작성 순서와 각 단계별 프롬프트 가이드 제공
> **대상 독자**: AI 코딩 에이전트(Claude Code 등)를 활용하여 프로젝트를 시작하는 개발자
> **예상 소요 시간**: 읽기 15분, 실습 적용 시 1-2시간
> **최종 수정일**: 2026-02-14

---

## 개요

AI 코딩 에이전트(Claude Code 등)를 활용하여 프로젝트를 시작할 때, **문서 작성 순서**가 매우 중요합니다. 잘못된 순서로 문서를 작성하면 기술 스택이 요구사항과 맞지 않거나, 구현 계획이 불완전해지는 등의 문제가 발생합니다.

이 가이드는 **올바른 문서 작성 순서**와 각 단계에서 사용할 **프롬프트 예시**를 제공합니다.

## ① PRD.md — 가장 먼저 (사람 주도)

### 역할

프로젝트의 **존재 이유**를 정의하는 최상위 문서입니다. 모든 후속 문서의 입력이 됩니다.

### 작성 주체

**사람** (또는 사람 + AI 협업)

### 포함 내용

- **프로젝트 배경과 목적**: 왜 이 프로젝트가 필요한가?
- **사용자 페르소나**: 누가 사용하는가? 각 사용자의 Pain Points는?
- **기능 요구사항 + 우선순위**: 무엇을 만드는가? (P0/P1/P2 우선순위)
- **비기능 요구사항**: 성능, 보안, 확장성 목표
- **데이터 모델**: 핵심 엔티티와 관계
- **마일스톤**: 언제까지 무엇을 완성하는가?
- **리스크 및 의존성**: 예상 위험 요소

### 프롬프트 예시 (AI와 협업 시)

```
다음 프로젝트의 PRD를 작성해줘:

[프로젝트 한 줄 설명]

핵심 기능:
- 기능 1: ...
- 기능 2: ...

사용자: ...
제약조건: ...

PRD에는 다음 섹션을 포함해줘:
1. 프로젝트 개요 (배경, 목적, 범위)
2. 목표 및 성공 지표
3. 사용자 및 페르소나
4. 기능 요구사항 (우선순위 P0/P1/P2 포함)
5. 비기능 요구사항 (성능, 보안, 확장성)
6. 데이터 모델 (핵심 엔티티와 관계)
7. 시스템 아키텍처 (권장 기술 스택 - 기술 중립적으로)
8. 마일스톤 및 로드맵
9. 리스크 및 의존성
10. 부록 (용어 정의, 우선순위 정의)
```

### 핵심 원칙

> ⚠️ **PRD는 기술 중립적이어야 합니다.**

**❌ 나쁜 예:**
```
"FastAPI를 사용한다"
```

**✅ 좋은 예:**
```
"대량 데이터 처리가 가능한 백엔드"
```

기술 선택은 다음 단계에서 PRD 요구사항에 맞춰 결정합니다. PRD에서 "권장 기술 스택"을 제시하는 것은 괜찮지만, 확정은 ②번 단계에서 합니다.

## ② 기술 분석 문서 — PRD 직후 (AI 주도)

### 역할

PRD 요구사항을 기술적으로 분석하고 **스택을 확정**하는 문서입니다. 이 단계를 건너뛰면 기술 스택이 요구사항과 맞지 않는 문제가 발생합니다.

### 작성 주체

**AI** (PRD를 입력으로)

### 포함 내용

- **PRD 요구사항별 기술적 복잡도 평가**: 각 기능이 얼마나 어려운가?
- **기술 스택 비교 및 선택 근거**: 최소 2-3개 후보를 비교하고 선택 이유를 PRD 요구사항 ID와 매핑
- **성능 목표 달성 전략**: PRD의 성능 요구사항별로 구체적인 기술적 방안
- **자동화 가능/불가 영역 분류**: 어떤 부분이 AI로 자동 생성 가능하고, 어떤 부분이 수동 구현 필수인지
- **리스크 분석**: 기술적 위험 요소

### 프롬프트 예시

> 💡 **프롬프트 팁**: `@파일명` 표기는 Claude Code에서 파일을 컨텍스트로 로드하는 문법입니다. 파일 경로를 직접 언급하거나 파일 내용을 붙여넣어도 됩니다.

```
@PRD.md 를 분석하여 다음을 수행해줘:

1. **기술 스택 분석**: PRD의 기능/비기능 요구사항을 충족하는
   최적의 기술 스택을 비교 분석해줘.
   - Backend 프레임워크 (최소 3개 비교)
   - Frontend 프레임워크 (최소 2개 비교)
   - Database
   - 대규모 데이터 처리 전략 (해당되는 경우)

   각 선택에 대해 "선택 근거"를 PRD 요구사항 ID(F1.1 등)와 매핑해줘.

2. **자동화 분석**:
   - 어떤 부분이 AI로 자동 생성 가능한지 (보일러플레이트, CRUD 등)
   - 어떤 부분이 수동 구현 필수인지 (도메인 로직, 알고리즘)
   - 각 Phase별 자동화 비율 추정

3. **성능 달성 전략**:
   - PRD의 각 성능 요구사항별로 구체적인 기술적 달성 방안 제시
   - 병목 지점 식별
   - Phase별 성능 목표

4. **기술 스택 확정**:
   - Backend, Frontend, Database, Cache, Batch Processing 등
   - 각 선택의 근거를 PRD 요구사항과 연결

결과물은 TECH-ANALYSIS.md 또는 AUTOMATION-ANALYSIS.md로 저장해줘.
출력은 한국어로 해줘.
```

### 산출물 예시

- `TECH-ANALYSIS.md`: 기술 스택 비교 및 확정
- `AUTOMATION-ANALYSIS.md`: 자동화 가능/불가 분석
- `Feasibility_Study.md`: 사업 타당성 분석 (선택 사항)

## ③ CLAUDE.md — 기술 분석 후 (AI 주도, 사람 검토)

### 역할

AI 코딩 에이전트의 **작업 가이드**입니다. 프로젝트에서 가장 중요한 문서이며, 모든 에이전트가 이 문서를 참조합니다.

### 작성 주체

**AI** (PRD + 기술 분석을 입력으로) → **사람이 검토/수정**

### 포함 내용

- **확정된 기술 스택**: 버전까지 명시, 선택 근거 포함
- **서브에이전트 정의**:
  - 각 에이전트의 역할, 책임, 산출물
  - 에이전트 간 의존성 그래프
  - 핸드오프 프로토콜
- **코딩 규칙**:
  - 절대 규칙 (위반 시 버그 발생하는 것)
  - 권장 패턴 / 금지 패턴
  - 구체적 코드 예시
- **작업 흐름**: Phase별 작업 순서, 검증 방법
- **문서 참조**: PRD, 기술 분석 등 관련 문서 경로

### 프롬프트 예시

```
@PRD.md 와 @TECH-ANALYSIS.md 를 기반으로 CLAUDE.md를 작성해줘.

CLAUDE.md는 이 프로젝트에서 Claude Code가 작업할 때
참조하는 가이드입니다. 다음 섹션을 포함해줘:

1. **프로젝트 개요**:
   - PRD에서 핵심 목표, 제약사항 요약 (3-5개 bullet points)
   - 중요한 제약사항 강조 (정확도, 성능, 보안 등)

2. **확정 기술 스택**:
   - 기술 분석에서 확정된 스택 (표 형식)
   - 버전 명시
   - 각 선택의 근거 간략히 포함

3. **서브에이전트 정의**:
   - 각 에이전트의 역할, 책임, 산출물 범위
   - 에이전트 간 의존성 그래프
   - 핸드오프 프로토콜 (API contract, 코드 리뷰 규칙)
   - 각 에이전트별 코드 스타일 예시 포함

4. **코딩 규칙**:
   - 절대 규칙 (위반 시 버그 발생하는 것, 예: Decimal 사용 필수)
   - 권장 패턴 / 금지 패턴 (구체적 코드 예시 포함)
   - 에러 핸들링 표준
   - 감사 로그 패턴 (해당되는 경우)

5. **작업 흐름**:
   - Phase별 작업 순서 (Phase A, B, C 등)
   - 각 Phase의 완료 기준 및 검증 방법

6. **문서 참조**:
   - PRD, 기술 분석 등 관련 문서 경로

중요:
- 서브에이전트 하나당 책임은 최대 5개로 제한
- 코딩 규칙에서 모호한 표현 없이 구체적 코드 예시 포함
- 금지 패턴은 "왜 금지인지" 이유도 포함
- 출력은 한국어로 해줘
```

### 핵심 원칙

> ⚠️ **CLAUDE.md는 PRD의 파생물이어야 합니다.**

❌ PRD 없이 CLAUDE.md를 먼저 만들면:
- 기술 스택이 요구사항과 불일치
- 중요한 제약사항(성능, 정확도 등)이 코딩 규칙에 누락
- 서브에이전트 역할이 비즈니스 요구사항과 맞지 않음

## ④ 서브에이전트 파일 생성 — CLAUDE.md 후 (AI 주도)

### 역할

CLAUDE.md에 정의된 서브에이전트 **역할을 구체화**하여 각 에이전트의 **작업 지침 파일**을 생성합니다. 이 파일들은 실제 개발 시 각 에이전트가 참조하는 상세 매뉴얼입니다.

### 작성 주체

**AI** (CLAUDE.md의 서브에이전트 정의를 입력으로)

### 저장 위치

`.claude/subagents/` 디렉토리

예시:
- `backend-developer.md`
- `frontend-developer.md`
- `devops-engineer.md`
- `qa-specialist.md`

### 포함 내용

각 서브에이전트 파일에는:
- **역할 요약**: 이 에이전트가 담당하는 영역 (1-2문장)
- **책임 범위**: 구체적으로 무엇을 하는가 (3-5개 항목)
- **작업 지침**:
  - 코딩 스타일 (들여쓰기, 네이밍 등)
  - 권장 패턴 / 금지 패턴
  - 에러 핸들링 표준
  - 로깅 규칙
- **산출물**: 생성해야 하는 파일 및 디렉토리 목록
- **검증 방법**: 작업 완료 기준 및 테스트 방법
- **참조 문서**: PRD, CLAUDE.md, API 문서 경로

### 프롬프트 예시

> 💡 **프롬프트 팁**: 서브에이전트 파일은 CLAUDE.md의 서브에이전트 정의를 **확장**하는 것입니다. 단순 복사가 아니라 실제 작업에 필요한 구체적인 지침을 추가합니다.

```
@CLAUDE.md 의 서브에이전트 정의를 기반으로
각 에이전트의 작업 지침 파일을 생성해줘.

생성할 서브에이전트:
- backend-developer: API, DB 스키마, 비즈니스 로직
- frontend-developer: UI 컴포넌트, 상태 관리, API 연동
- devops-engineer: Docker, CI/CD, 모니터링

각 파일은 다음을 포함해줘:
1. **역할 요약**: 1-2문장으로 이 에이전트의 책임 설명
2. **책임 범위**: 구체적인 작업 항목 3-5개
3. **작업 지침**:
   - 코딩 스타일 (들여쓰기, 네이밍 규칙)
   - 권장 패턴 예시 코드
   - 금지 패턴 및 이유
   - 에러 핸들링 표준
4. **산출물**: 생성할 파일 목록
5. **검증 방법**: 작업 완료 기준 (테스트, curl 명령 등)
6. **참조 문서**: PRD, CLAUDE.md 경로

파일은 .claude/subagents/ 디렉토리에 저장해줘.
각 파일에 CLAUDE.md의 코딩 규칙을 반영해야 해.
출력은 한국어로 해줘.
```

### 예시: backend-developer.md 구조

```markdown
# Backend Developer Agent

## 역할 요약
FastAPI 기반 백엔드 API 구현, DB 스키마 설계, 비즈니스 로직 개발을 담당합니다.

## 책임 범위
- RESTful API 엔드포인트 구현
- SQLAlchemy 모델 및 마이그레이션
- CRUD 로직 및 비즈니스 규칙
- API 문서 자동 생성 (Swagger)
- 단위 테스트 작성

## 작업 지침

### 코딩 스타일
- Python 3.12+ 타입 힌트 필수
- 함수명: snake_case
- 클래스명: PascalCase
- 상수: UPPER_CASE

### 권장 패턴
\`\`\`python
# 의존성 주입 패턴
async def get_items(
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    ...
\`\`\`

### 금지 패턴
❌ ORM 쿼리를 컨트롤러에 직접 작성 → CRUD 레이어로 분리

## 산출물
- `app/api/endpoints/*.py`: API 라우터
- `app/models/*.py`: SQLAlchemy 모델
- `app/schemas/*.py`: Pydantic 스키마
- `app/crud/*.py`: CRUD 로직
- `tests/test_*.py`: 테스트 파일

## 검증 방법
- `pytest tests/` 통과
- `curl http://localhost:8000/docs` → Swagger UI 접근
- 모든 API 응답 시간 < 200ms

## 참조 문서
- [PRD](../PRD.md) - 기능 요구사항
- [CLAUDE.md](../CLAUDE.md) - 코딩 규칙
```

### 핵심 원칙

> 💡 **서브에이전트 파일은 "실행 가능한 지침서"**

CLAUDE.md는 프로젝트 전체 가이드이고, 서브에이전트 파일은 **특정 역할에 필요한 지침만 추출**하여 구체적인 코드 예시와 함께 제공합니다.

✅ **좋은 서브에이전트 파일**:
- 코드 예시가 풍부함
- "하지 말아야 할 것"과 이유를 명시
- 검증 방법이 구체적 (curl 명령, 테스트 실행 등)

❌ **나쁜 서브에이전트 파일**:
- CLAUDE.md를 단순 복사
- 추상적인 설명만 있고 예시 없음
- 검증 방법이 모호함

## ⑤ 구현 계획서 — 서브에이전트 파일 생성 후 (AI 주도)

### 역할

실제 코딩 작업의 **단계별 실행 계획**입니다. "무엇을 언제 어떤 순서로 만들 것인가"를 정의합니다.

### 작성 주체

**AI** (PRD + CLAUDE.md를 입력으로)

### 포함 내용

- **Phase별 Step 목록**: 각 Step은 독립적으로 검증 가능한 단위
- **파일 생성 순서**: FK 의존성, import 의존성 고려
- **각 Step의 완료 기준**: "무엇이 작동하면 완료인가?"
- **검증 방법**: curl 명령, 테스트 실행 등 구체적 방법
- **Critical files 식별**: 잘못 구현 시 전체에 영향을 미치는 파일

### 프롬프트 예시

```
@PRD.md 와 @CLAUDE.md 를 기반으로 구현 계획을 작성해줘.

요구사항:
1. **Phase별로 구분**: 각 Phase는 독립적으로 검증 가능해야 함
   - Phase A: 프로젝트 골격 (CRUD, 인증, Docker)
   - Phase B: 핵심 비즈니스 로직
   - Phase C: 고도화 (시뮬레이션, 최적화)

2. **Step 단위 분할**: 각 Step은 "완료 후 즉시 테스트 가능한 단위"
   - 예: "Step 1: .gitignore + .env.example 생성"
   - 예: "Step 2: Backend 기본 구조 (main.py, config.py, database.py)"

3. **의존성 고려**:
   - 파일 생성 순서는 FK 의존성, import 의존성 고려
   - 예: base.py → models → schemas → crud → api

4. **검증 방법 명시**: 각 Phase 완료 후 검증 방법
   - 예: "curl http://localhost:8000/health → {\"status\": \"ok\"}"
   - 예: "docker compose up → 4개 서비스 모두 healthy"

5. **Critical files 식별**:
   - 잘못 구현 시 전체에 영향을 미치는 파일
   - 예: models/base.py, crud/base.py

결과물은 .claude/plans/IMPLEMENTATION_PLAN.md로 저장해줘.
출력은 한국어로 해줘.
```

### 산출물 저장 위치

`.claude/plans/IMPLEMENTATION_PLAN.md` (또는 프로젝트 루트)

## ⑥ README.md — 프로젝트 골격 완성 후 (최후)

### 역할

프로젝트 **사용 설명서**입니다. 개발자와 사용자가 프로젝트를 이해하고 사용하는 방법을 안내합니다.

### 작성 주체

**AI** (프로젝트 코드가 존재한 후)

### 포함 내용

- 프로젝트 소개 (1-2문장)
- 기술 스택
- 빠른 시작 (Quick Start)
- 개발 환경 설정
- 프로젝트 구조 (디렉토리 트리)
- API 문서 접근 방법
- 테스트 실행 방법

### 프롬프트 예시

```
현재 프로젝트 구조를 분석하여 README.md를 작성해줘.

포함할 내용:
1. **프로젝트 소개**: 한 줄 설명 + 핵심 기능 3-5개
2. **기술 스택**: 주요 기술 및 버전
3. **빠른 시작 (Quick Start)**:
   - Docker Compose: `docker compose up` 또는 `./run.sh start`
   - 로컬 실행 방법 (개발 모드)
4. **개발 환경 설정**:
   - 필수 요구사항 (Node, Python 버전 등)
   - 환경 변수 설정 (.env.example 참조)
5. **프로젝트 구조**: 디렉토리 트리 및 주요 디렉토리 설명
6. **API 문서**: Swagger UI 접근 방법 (http://localhost:8000/docs)
7. **테스트 실행**: pytest, Jest 등
8. **기여 가이드** (선택 사항)

출력은 한국어로 해줘.
```

### 작성 시기

**Phase A (프로젝트 골격)가 완성된 후**, 즉 Docker로 서비스가 실행 가능한 상태가 된 후에 작성합니다.

## 실전 예시: 문서 순서를 건너뛴 경우의 문제

### 잘못된 순서

```
① PRD.md 작성            ✅ 올바름
② 기술 분석 건너뜀        ❌
③ CLAUDE.md 바로 작성    ❌ 요구사항 분석 없이 기술 스택 결정
④ 구현 계획              ❌ 불완전한 CLAUDE.md 기반
⑤ 코드 생성              ❌ 검증 안 된 계획으로 개발
⑥ README.md              ⚠️ 시기는 맞으나 내용 부실
```

### 발생 가능한 문제

1. **기술 스택 불일치**: PRD에서 권장한 기술(예: 대량 데이터 처리 도구)이 기술 분석 없이 CLAUDE.md에 누락되거나 다른 기술로 대체됨
2. **코딩 규칙 혼재**: PRD에서 명확히 정의한 표준(예: 숫자 처리 방식, 날짜 형식 등)이 CLAUDE.md에서 여러 방법으로 혼재되어 일관성 부족
3. **서브에이전트 책임 과부하**: 체계적 분석 없이 서브에이전트를 정의하여 한 에이전트가 7개 이상의 책임을 갖게 되는 불균형 발생
4. **성능 목표 미달**: PRD의 비기능 요구사항(처리량, 응답시간 등)을 달성할 구체적인 기술적 방안이 CLAUDE.md와 구현 계획에 반영되지 않음

### 올바른 순서로 수정하기

```
① PRD.md (이미 존재)        ✅
② 기술 분석 수행            ← 먼저 수행
③ CLAUDE.md 개정            ← 기술 분석 기반으로 수정
④ 서브에이전트 파일 생성    ← CLAUDE.md 기반으로 구체화
⑤ 구현 계획 수정            ← 새로운 CLAUDE.md + 서브에이전트 기반으로
⑥ README.md 업데이트        ← Phase A 완성 후
```

> 💡 **교훈**: 문서 작성 순서를 건너뛰면 후속 문서의 품질이 떨어지고, 개발 중 반복적인 수정이 필요하여 전체 개발 시간이 오히려 증가합니다.

## 체크리스트

### PRD.md 작성 시

- [ ] 프로젝트의 "왜"가 명확한가?
- [ ] 사용자 페르소나와 Pain Points가 구체적인가?
- [ ] 기능 요구사항에 우선순위(P0/P1/P2)가 있는가?
- [ ] 비기능 요구사항(성능, 보안)이 **숫자로 명시**되어 있는가?
- [ ] 기술 스택이 "권장" 수준이고, 확정이 아닌가?

### 기술 분석 문서 작성 시

- [ ] PRD의 모든 성능 요구사항에 대해 달성 전략이 있는가?
- [ ] 기술 스택 비교에서 최소 2-3개 후보를 비교했는가?
- [ ] 각 기술 선택의 근거가 PRD 요구사항 ID와 매핑되어 있는가?
- [ ] 자동화 가능/불가 영역이 명확히 분류되어 있는가?

### CLAUDE.md 작성 시

- [ ] 기술 스택이 기술 분석에서 확정된 것과 일치하는가?
- [ ] 서브에이전트 하나당 책임이 5개 이하인가?
- [ ] 코딩 규칙에 구체적인 코드 예시가 있는가?
- [ ] PRD의 제약사항(정확도, 성능)이 코딩 규칙에 반영되어 있는가?
- [ ] 에이전트 간 의존성과 핸드오프 프로토콜이 정의되어 있는가?

### 서브에이전트 파일 생성 시

- [ ] 각 서브에이전트 파일이 `.claude/subagents/`에 생성되었는가?
- [ ] 역할 요약이 명확하고 간결한가? (1-2문장)
- [ ] 작업 지침에 구체적인 코드 예시가 포함되어 있는가?
- [ ] 권장 패턴과 금지 패턴이 "왜"와 함께 설명되어 있는가?
- [ ] 검증 방법이 실행 가능한 명령어로 명시되어 있는가?
- [ ] CLAUDE.md의 코딩 규칙이 각 에이전트 파일에 반영되어 있는가?

### 구현 계획 작성 시

- [ ] 각 Step이 독립적으로 검증 가능한가?
- [ ] 파일 생성 순서가 의존성(FK, import)을 고려했는가?
- [ ] 각 Phase 완료 후 검증 방법이 구체적인가?
- [ ] Critical files가 식별되어 있는가?

## 요약: 한 줄 정리

```
PRD.md (무엇) → 기술 분석 (어떤 도구로) → CLAUDE.md (어떻게) → 서브에이전트 파일 (누가) → 구현 계획 (순서) → 코드 → README.md
```

> 💡 **핵심 메시지**: 각 문서는 이전 문서를 입력으로 받아 생성되므로, 순서를 건너뛰면 이후 문서의 품질이 떨어집니다.

⚠️ **주의**: 시간이 없다고 단계를 건너뛰면, 나중에 문서를 다시 작성하느라 더 많은 시간이 소요됩니다.
